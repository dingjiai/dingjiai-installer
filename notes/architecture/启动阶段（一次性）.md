# 启动阶段（一次性）

本文只讨论 Windows-first 安装器的**启动阶段（一次性）**。

这里的目标不是安装任何业务组件，而是把机器和本地运行环境整理到一个**可安全进入主流程**的状态，并最终把控制权成功交给新的管理员 `cmd.exe` 主窗口。

本文是启动阶段的详细规范源。`notes/architecture/整体架构.md` 只保留启动阶段摘要、分层位置和对本文的引用；如果两份文档出现细节差异，启动阶段的字段、顺序、状态、handoff、payload 和安全边界以本文为准，整体架构文档应回写同步摘要，避免双写漂移。

## 一、启动阶段的唯一目标

本轮讨论明确否定“系统 / 网络 / 基础能力”这种泛泛表述。

启动阶段的每一项检查都必须落到：

- 具体检查字段
- 具体失败处理
- 具体边界
- 明确不做什么

也就是说，不能只写“检查系统、检查网络、检查基础能力”，而要明确到底检查什么、为什么现在检查、失败后如何处理。

启动阶段只负责四件事：

1. 确认当前机器**可以安全进入安装器主流程**。
2. 准备好本地 launcher payload。
3. 成功切换到新的管理员 `cmd.exe` 主窗口。
4. 把控制权交给主菜单。

启动阶段的唯一成功标准是：

> **新的管理员 `cmd.exe` 主窗口已成功启动，并接管安装器主入口。**

只要某个关键检查没有通过，就算启动失败。

本项目采用逐层建设方式，但每一层写完就必须达到该层应有的硬度。启动框架本身不能因为业务逻辑尚未接入而留下已知脆弱点；后续业务模块也应按同样原则推进：写一个，硬一个，避免把基础质量债留到后期返工。

## 二、为什么启动阶段要严格借鉴 MAS

本项目的启动阶段应当把 `massgravel/Microsoft-Activation-Scripts`（MAS）作为默认机制参考。

这里的含义不是“参考一下即可”，而是：

> **启动阶段的兼容性收敛、宿主修正、权限处理、重入保护和 handoff 思路，默认应优先按 MAS 经过多年迭代验证过的机制来设计。**

原因是启动阶段不是业务创新区，而是兼容性高风险区。

这一层涉及：

- 不同 Windows 版本
- 不同 PowerShell 宿主
- 32 位 / 64 位进程差异
- UAC 提权
- 新进程启动
- 远程脚本执行
- 用户从各种入口壳启动
- 失败后的重入与退出

这些问题需要长期验证、试错和迭代。MAS 已经在类似入口场景里经历过多年真实用户环境检验，因此本项目不应在启动阶段重复造轮子。

借鉴重点不是照搬其激活业务，也不是复制 MAS 的 GPL-3.0 代码，而是学习它这些年沉淀下来的**启动兼容性处理机制**，并用 MIT 兼容的新代码重新实现。

MAS 在启动阶段最值得学习的原则有：

当前已确认：启动阶段后续就按照这个方向继续细化，不再另起一套启动机制。

定稿方向是：

```text
启动阶段严格参考 MAS 的成熟启动机制；
除非 MAS 的做法和本项目“payload + 管理员 CMD 主窗口”目标冲突，
否则不自行发明新机制。
```

具体落地时应按 MAS 的成熟模式拆解并映射到本项目：

```text
极薄入口
-> 宿主归一化
-> 位数收敛
-> 权限收敛
-> 重入保护
-> 平台硬门槛尽早失败
-> payload 获取与校验
-> 管理员 CMD handoff
```

实现要求仍然是：

```text
学习机制
重新实现
不复制代码
保持 MIT 兼容
```

MAS 在启动阶段最值得学习的原则有：

1. **先把当前宿主修正到正确形态**。
   - 例如 32 位进程跑在 64 位系统上时，先切到正确宿主。
   - 特殊环境不支持时，尽早停止。
2. **先解决权限，再谈后续功能**。
   - 不够权限就提权。
   - 提权失败就明确退出。
3. **先卡死最低平台门槛**。
   - 版本不够、能力不足，就不要拖进主流程。
4. **用“重启自身”做兼容性收敛**。
   - 不是在错误宿主里硬扛兼容，而是切到正确宿主后重新进入流程。
5. **必须有重入保护，防止无限循环**。

因此，本项目在启动阶段的总原则是：

> **先归一化运行环境，再进入主逻辑。**

## 三、MAS 机制映射：极薄入口

### 1. MAS 的成熟做法

MAS 的远程入口形态非常薄。

典型入口是：

```powershell
irm https://get.activated.win | iex
```

它的价值不是把所有业务都塞进入口命令里，而是让用户用最低摩擦进入脚本体系。

MAS 的入口层承担的是：

```text
降低进入成本
-> 拉起脚本
-> 进入后续受控流程
```

而不是：

```text
在入口阶段完成所有业务判断
在入口阶段做完整系统巡检
在入口阶段承担长期交互 UI
```

这就是“极薄入口”的价值。

### 2. 本项目的对应入口

本项目当前 Windows 入口是：

```powershell
irm https://get.dingjiai.com/win.ps1 | iex
```

这个入口也应该保持极薄。

它只应该负责：

1. 让用户从任意支持的 PowerShell 入口进入 bootstrap。
2. 做最小启动检查。
3. 准备本地 payload。
4. handoff 到新的管理员 `cmd.exe` 主窗口。

入口本身不应该成为主程序。

### 3. 极薄入口明确不做什么

入口层不做：

- 不显示正式主菜单。
- 不执行安装 / 更新 / 卸载。
- 不检测 Git / Claude / gh / rg / Python 等业务组件。
- 不检查 GitHub、npm、winget 等业务下载源。
- 不写用户配置。
- 不修 PATH。
- 不承担长期交互。

这些都应发生在本地主程序接管之后。

### 4. 为什么入口必须薄

入口越厚，兼容性风险越高。

因为入口运行在用户随手打开的原始宿主里，可能是：

- Windows Terminal
- PowerShell 5.1
- PowerShell 7
- 32 位 PowerShell
- 管理员 PowerShell
- 非管理员 PowerShell
- 被策略限制的 PowerShell
- 代理 / TLS / 编码环境不一致的 PowerShell

这一层越多业务逻辑，就越容易踩到宿主差异。

因此入口阶段应该尽快完成：

```text
进入 -> 校验能继续 -> 准备 payload -> 交给稳定主宿主
```

### 5. 极薄入口的成功标准

入口层自己的成功标准不是“安装器完成”，而是：

```text
可信本地 payload 已准备好，且管理员 cmd.exe 主窗口已开始接管。
```

如果无法做到这一点，就应该明确启动失败，而不是在原始 PowerShell 中降级运行正式菜单。

### 6. 和 MAS 的差异

MAS 的入口最终进入的是它自己的 CMD 脚本体系。

本项目的入口最终进入的是：

```text
本地 payload
+ 新开的管理员 cmd.exe 主窗口
+ 本地主菜单入口
```

所以本项目不是复制 MAS 的入口代码，而是复用 MAS 的入口思想：

```text
远程入口极薄
真实逻辑后移
宿主先收敛
主流程在稳定宿主中运行
```

## 四、启动阶段的五段式结构

当前讨论建议把启动阶段固定为五段：

```text
A. 宿主归一化
B. 平台硬门槛
C. 本地工作区
D. payload 获取与校验
E. handoff 到管理员 CMD
```

其中：

- **A 和 E 最像 MAS 的成熟实践**。
- **D 是本项目相对 MAS 更需要重点设计的部分**，因为本项目是“拉取并落地本地主程序”型启动器。

审阅后统一口径如下：

```text
A. 宿主归一化负责修正当前 bootstrap 宿主形态，位数收敛属于 A 的子回环。
B. 平台硬门槛只检查是否具备继续启动的硬条件，不执行 UAC handoff。
C. 本地工作区负责目录、锁、状态和会话基座，不负责判断 payload 内容可信。
D. payload 获取与校验负责 manifest、版本一致性、hash、签名或信任锚、重下和缓存容灾边界。
E. handoff 到管理员 CMD 才执行实际 UAC 提权、参数传递和接管确认。
```

也就是说，`cmd.exe` 和 `RunAs` 能力可以在 B 阶段被检查，但真正打开管理员 `cmd.exe`、写入 `handoffAttempted`、等待 `handoffAccepted`，只能发生在 E 阶段。

## 五、启动阶段详细拆分

### A. 宿主归一化

这一步是最应该严格借鉴 MAS 的地方。

MAS 的成熟点不是在各种不稳定宿主里继续硬兼容，而是：

```text
先判断当前宿主是否适合继续
-> 如果不适合，切到更正确的宿主
-> 带重入标记重新进入脚本
-> 再次检查
-> 仍不合格则失败退出
```

本项目也应采用这个模式。

目标：把 bootstrap 当前所处的运行壳收敛到一个**正确且可控**的形态。

#### 宿主归一化的判断字段

这一段建议只检查和“当前脚本能不能继续启动”有关的字段：

1. 当前是否运行在 Windows。
2. 当前是否运行在 PowerShell 语义中。
3. 当前 PowerShell 版本是否满足 bootstrap 最低要求。
4. 当前 OS 是否为 64 位。
5. 当前进程是否为 64 位。
6. 如果是 64 位 OS + 32 位进程，是否能切换到 64 位 PowerShell。
7. 当前宿主是否能发起 HTTPS 请求。
8. 当前宿主是否能写入临时位置。
9. 当前宿主是否能启动新进程。
10. 当前宿主是否能调用 `cmd.exe`。
11. 当前宿主是否能发起 `Start-Process -Verb RunAs`。

这些字段只服务于启动阶段，不包含业务组件发现。

#### 宿主合格的最低标准

启动阶段不要求当前入口壳成为长期 UI 宿主。

它只要求当前入口壳足够完成 bootstrap：

```text
能运行当前脚本
能下载 / 校验本项目 payload
能启动新进程
能发起管理员 CMD handoff
```

一旦 handoff 成功，后续长期交互不再依赖这个原始入口壳。

#### 32 位到 64 位的收敛

如果发现：

```text
64 位 Windows
+ 当前 PowerShell 是 32 位进程
```

则不应继续在 32 位进程里跑后续启动逻辑。

应当按 MAS 风格切换到 64 位 PowerShell 后重新进入启动流程。

原因是 32 位进程会带来：

- `System32` / `SysWOW64` 路径重定向
- 系统工具解析差异
- 注册表视图差异
- 后续 `cmd.exe` / 安装器 / PATH 判断不一致

因此，当前建议：

```text
64 位 Windows 上，启动阶段应尽早收敛到 64 位进程语义。
```

#### PowerShell 宿主选择原则

本项目入口可以从不同 PowerShell 形态进入，例如：

- Windows PowerShell 5.1
- PowerShell 7 `pwsh`
- Windows Terminal 中的 PowerShell
- 管理员或非管理员 PowerShell

但启动阶段不应把这些入口壳都当成长期宿主。

判断原则应是能力优先：

```text
当前宿主能力足够 -> 继续 bootstrap
当前宿主位数或能力不合格 -> 切换到系统内更合适的 PowerShell 宿主后重入
```

在 Windows-first v1 中，64 位 Windows PowerShell 5.1 通常更适合作为保底 bootstrap 宿主，因为它是系统内置能力；PowerShell 7 不应成为启动阶段的硬依赖。

#### 前置重入标记

宿主归一化发生得非常早。

此时本地工作区可能还没创建，完整的 `startup-<startupId>.json` 也可能还不可用。

因此启动阶段需要区分两类标记：

```text
前置重入标记：用于宿主归一化之前或过程中
正式状态文件：用于本地工作区建立之后
```

前置重入标记可以通过：

- 命令行参数
- 环境变量
- 临时最小 marker

来传递。

它至少要表达：

```text
已经尝试过宿主归一化
不要无限重启自己
```

前置重入标记不是完整状态源，只是正式状态文件创建前的防循环桥接信息。

统一规则是：

```text
startupId 尽早生成，并贯穿前置重入标记与正式状态文件。
工作区尚未建立前，只允许读取前置标记判断是否还能重入。
工作区建立后，必须把前置标记迁移进 startup-<startupId>.json。
迁移完成后，正式状态文件成为唯一真相源。
如果前置标记与正式状态文件冲突，按更保守的结果处理：已经尝试过的动作不得再尝试。
```

本地工作区准备完成后，再把这些信息写入正式启动状态文件。

#### 宿主归一化的流程

建议流程：

```text
进入 bootstrap
-> 检查 Windows
-> 检查 PowerShell 语义
-> 检查 PowerShell 最低版本
-> 检查 OS / 进程位数
-> 如果 64 位 OS + 32 位进程：
      若未尝试过归一化：
          切到 64 位 PowerShell 并带标记重入
      若已尝试过仍不合格：
          失败退出
-> 检查 HTTPS / 写临时文件 / 启动进程 / RunAs 基础能力
-> 通过后进入平台硬门槛
```

#### 失败处理

宿主归一化失败时，不进入主菜单，也不进入 payload 业务。

失败时应说明：

- 当前失败阶段是宿主归一化。
- 失败原因是宿主能力不足、位数无法修正，还是 PowerShell 不满足要求。
- 用户可以重新打开系统自带 PowerShell 后再运行一行命令。

#### 这一段绝对不做的事

- 不检查 Git
- 不检查 Claude
- 不检查 winget
- 不做 GitHub / npm / winget 下载源巡检
- 不做业务层发现
- 不显示正式菜单

宿主归一化一定要发生在最前面。

### A-1. MAS 机制映射：位数收敛

位数收敛是 MAS 启动机制里非常关键的一类兼容性处理。

Windows 上经常出现这种情况：

```text
系统是 64 位 Windows
但当前脚本运行在 32 位 PowerShell / 32 位 cmd 进程里
```

这不是一个小问题。

32 位进程在 64 位 Windows 上会遇到：

- `System32` 被重定向到 `SysWOW64`
- 系统工具路径解析不同
- 注册表视图不同
- 某些系统命令和组件探测结果不同
- 后续安装器、PATH、活动命令解析可能和真实用户环境不一致

MAS 的成熟处理思路不是在 32 位进程里补一堆特殊判断，而是：

```text
发现 64 位系统 + 32 位进程
-> 切换到 64 位宿主
-> 重新进入脚本
-> 重新检查
```

也就是把问题前移解决。

#### 本项目为什么必须做位数收敛

本项目后续会涉及：

- `cmd.exe` handoff
- `winget`
- Git for Windows
- Claude CLI
- PATH 检测与修复
- 新 shell 验证
- 系统目录和用户目录解析

如果启动阶段停留在错误位数进程里，后续所有检测都可能变得不可靠。

因此本项目应确认：

```text
64 位 Windows 上，启动阶段必须尽早收敛到 64 位进程语义。
```

#### 位数收敛的判断字段

建议至少判断：

1. OS 是否为 64 位。
2. 当前 PowerShell 进程是否为 64 位。
3. 当前是否能定位 64 位 PowerShell。
4. 当前是否已经尝试过位数收敛。
5. 重入后是否仍然是 32 位进程。

其中第 4 点必须依赖前置重入标记，而不是等正式状态文件创建后才判断。

#### 位数收敛的目标宿主

在 Windows-first v1 中，位数收敛的目标不应依赖用户是否安装了 PowerShell 7。

更稳妥的目标是系统自带的 64 位 Windows PowerShell。

也就是说：

```text
优先收敛到系统内置 64 位 Windows PowerShell
而不是要求用户存在 pwsh
```

原因是 PowerShell 7 不是所有机器都有，而 Windows PowerShell 5.1 在支持范围内的 Windows 上更适合作为保底 bootstrap 宿主。

#### 位数收敛流程

建议流程：

```text
检查 OS 位数
-> 如果 OS 不是 64 位：
      根据项目支持范围决定是否失败
-> 如果 OS 是 64 位：
      检查当前进程位数
-> 如果当前进程已经是 64 位：
      继续后续启动检查
-> 如果当前进程是 32 位：
      检查是否已经尝试过位数收敛
      -> 未尝试：启动 64 位 PowerShell，带前置重入标记重新进入
      -> 已尝试仍是 32 位：失败退出
```

#### 位数收敛和重入保护

位数收敛必须是有限动作。

建议规则：

```text
64 位 OS 上从 32 位进程切换到 64 位进程：最多尝试 1 次
```

如果重入后仍然不满足目标，就不要继续绕路。

失败时应明确记录：

```text
阶段：host-normalize
失败原因：无法从 32 位宿主收敛到 64 位宿主
建议动作：请使用系统自带 PowerShell 重新运行入口命令
```

#### 位数收敛和业务逻辑边界

位数收敛只解决启动宿主形态问题。

它不做：

- 不判断 Git 是 32 位还是 64 位
- 不判断 Claude 安装位置
- 不判断 winget 包架构
- 不修 PATH
- 不做任何组件安装

这些全部属于后续组件 checkpoint。

#### 本项目和 MAS 的差异

MAS 的位数收敛服务于它自己的 CMD / 激活脚本体系。

本项目的位数收敛服务于：

```text
PowerShell bootstrap 能稳定准备 payload
并能可靠 handoff 到管理员 cmd.exe 主窗口
```

也就是说，机制参考 MAS，但目标改成本项目的主战场。

### B. 平台硬门槛

当前 v1 启动阶段硬门槛固定为 Windows build 17763+ 与 PowerShell 5.1+；不满足时必须在 workspace、manifest、payload、handoff 之前失败退出。

平台硬门槛是 MAS 风格启动机制中的“早失败”部分。

它的目标不是做完整系统巡检，而是尽早识别那些后面一定无法成功的问题。

MAS 的成熟原则是：

```text
不支持就早点停
不要把用户拖进一个后面一定会炸的流程
```

本项目应采用同样原则。

#### 平台硬门槛的边界

平台硬门槛只判断：

```text
这台机器是否具备启动本项目主程序的最低条件
```

不判断：

```text
这台机器是否一定能完成所有业务安装
```

因此，GitHub、npm、winget、Git、Claude 等业务层问题不应放在这里。

#### 需要检查的内容

建议检查：

1. Windows 版本是否在支持范围内。
2. 系统架构是否在支持范围内。
3. 64 位进程语义是否已经满足或已无法再收敛。
4. `cmd.exe` 是否可用。
5. 是否支持 `Start-Process -Verb RunAs`。
6. 本地工作区根目录是否可写。
7. 临时目录是否可写。
8. 是否允许启动新进程。
9. 是否具备基本 HTTPS / TLS 能力用于获取本项目 payload。

这些检查只服务于启动阶段。

#### 权限能力检查与实际 handoff 的边界

平台硬门槛阶段可以检查：

```text
cmd.exe 是否可用
Start-Process -Verb RunAs 是否可用
是否允许启动新进程
```

但这些只是能力门槛，不代表此时已经执行提权。

实际 UAC handoff 的执行点只能在 E 阶段：

```text
payload 已完整校验
-> handoff 参数已安全组装
-> 写入 handoffAttempted
-> 发起管理员 cmd.exe
-> 等待 handoffAccepted
```

因此，B 阶段的结论只能是“机器具备 handoff 能力”或“不具备 handoff 能力”；不能在 B 阶段打开管理员 CMD，也不能在 payload 尚未可信前发起 UAC。

#### 可收敛问题与不可收敛问题

平台硬门槛要先区分两类问题。

可收敛问题：

```text
当前状态不合格，但安装器可以安全修正
```

例如：

- 当前进程位数不对，但能切到 64 位宿主。
- 工作区目录不存在，但可以创建。
- payload 目录残缺，但可以清理重建。

不可收敛问题：

```text
当前状态不合格，且安装器不能安全修正
```

例如：

- 不是 Windows。
- Windows 版本低于最低支持范围。
- 系统架构不支持。
- `cmd.exe` 不可用。
- `Start-Process -Verb RunAs` 不可用。
- 用户拒绝 UAC。
- 本地工作区不可写且无法修复。

不可收敛问题应直接失败，不进入主菜单。

#### 失败处理原则

平台硬门槛失败时，应输出简单明确的信息。当前统一失败输出必须包含三部分：原因、建议、日志路径；如果失败发生在日志目录创建前，则明确说明尚未生成日志文件。

平台硬门槛失败时，应输出简单明确的信息：

```text
启动失败：当前 Windows 版本不受支持。
建议：请在支持的 Windows 版本上重新运行安装器。
```

或者：

```text
启动失败：无法启动管理员 CMD 主窗口。
建议：请确认没有拒绝 UAC 提权，然后重新运行入口命令。
```

失败信息要面向小白用户，不要直接堆内部异常。

#### 不应在平台门槛里检查的内容

平台硬门槛不检查：

- Git 是否安装。
- Claude 是否安装。
- gh / rg / Python 是否安装。
- GitHub 是否可访问。
- winget source 是否健康。
- npm registry 是否可访问。
- 用户 API key 是否存在。
- `settings.json` 是否存在。

这些属于安装阶段、source preflight checkpoint、组件 checkpoint 或产品配置阶段。

#### 平台硬门槛与 MAS 的对应关系

MAS 在进入主功能前会尽早排除不支持的系统状态。

本项目也要复现这个原则：

```text
先确认平台能启动主程序
再进入管理员 CMD 主窗口
再显示菜单
再进入安装 / 更新 / 卸载
```

如果平台门槛未过，启动阶段就应该结束。

### B-1. MAS 机制映射：权限收敛

权限收敛是 MAS 启动机制的另一个核心成熟点。

MAS 的原则是：

```text
先解决权限
再进入主功能
```

它不会把用户带进菜单或业务流程后，再让某个深层操作因为权限不足而失败。

本项目也应该采用同样原则，但目标宿主不同。

MAS 的提权通常是为了让脚本自身在管理员权限下继续执行；本项目的提权目标是：

```text
打开新的管理员 cmd.exe 主窗口
并让它接管本地主入口
```

也就是说，本项目不是让原始 PowerShell 提权后继续作为主 UI，而是通过 UAC 把控制权交给稳定的管理员 CMD 主战场。

#### 权限收敛的判断字段

建议至少判断：

1. 当前进程是否已经是管理员权限。
2. 当前是否能发起 UAC 提权。
3. `Start-Process -Verb RunAs` 是否可用。
4. `cmd.exe` 是否可用。
5. handoff 命令是否能被正确组装。
6. 是否已经尝试过 handoff。
7. 新管理员 CMD 是否写回接管标记。

这些字段的目标不是进入安装业务，而是确认能不能完成管理员 CMD handoff。

#### 权限收敛的基本流程

建议流程：

```text
完成宿主归一化
-> 准备本地 payload
-> 检查本地主入口存在
-> 检查 cmd.exe 可用
-> 检查 RunAs 可用
-> 组装 handoff 命令
-> 如果当前不是管理员主窗口：
      通过 UAC 启动管理员 cmd.exe
      等待或轮询接管标记
-> 新 cmd.exe 进入本地主入口
-> 新 cmd.exe 写回 handoffAccepted
-> 原始 PowerShell 确认接管后退出
```

这里要强调：

```text
UAC 提权命令发出成功 ≠ 启动阶段成功
cmd.exe 窗口出现 ≠ 启动阶段成功
新管理员 CMD 写回接管标记才算成功
```

#### handoff 接管标记

新管理员 CMD 进入本地主入口后，应写回启动状态：

```json
{
  "stage": "completed",
  "handoffAccepted": true
}
```

原始 PowerShell 看到这个标记后，才认为启动阶段成功。

如果没有看到该标记，则应该把结果视为失败、超时或未知，不能在原始窗口降级显示正式主菜单。

#### 用户拒绝 UAC 的处理

用户拒绝 UAC 不应被包装成普通错误继续重试。

它属于明确的用户取消：

```text
用户拒绝 UAC
-> handoff 失败
-> 原始窗口显示简短原因
-> 写日志
-> 退出
```

不应做：

```text
继续在非管理员 PowerShell 里显示正式菜单
继续尝试安装任务
无限弹 UAC
```

#### 权限收敛次数

权限收敛必须有限。

建议规则：

```text
管理员 CMD handoff：最多 1 次正式尝试
```

如果失败，用户可以重新运行入口命令。

启动器不应在同一次启动中反复弹出 UAC。

#### 已经是管理员时怎么办

如果用户一开始就在管理员 PowerShell 中运行入口命令，也不意味着原始 PowerShell 应该成为长期主 UI。

本项目的主战场仍然是：

```text
新的管理员 cmd.exe 主窗口
```

因此即使当前 PowerShell 已经是管理员，也应根据产品架构决定是否仍然 handoff 到专用 CMD 主窗口。

建议原则：

```text
管理员权限是必要条件，但不是主宿主条件。
```

也就是说：

```text
当前已管理员
-> 可以减少 UAC 阻力
-> 但仍要进入专用管理员 CMD 主窗口
```

#### 权限收敛和业务逻辑边界

权限收敛不做：

- 不开始安装 Git / Claude / gh
- 不修 PATH
- 不写用户配置
- 不做组件健康检查
- 不显示正式功能菜单

它只解决：

```text
能否进入项目定义的管理员 CMD 主战场
```

#### 本项目和 MAS 的差异

MAS 的权限收敛目标是让 MAS 脚本以合适权限继续运行。

本项目的权限收敛目标是：

```text
让本地 payload 在新的管理员 cmd.exe 主窗口中接管
```

所以本项目复现 MAS 的机制，但不照搬目标形态。

### B-2. MAS 机制映射：重入保护

重入保护是 MAS 风格启动机制能成立的前提。

MAS 这类脚本经常需要：

```text
发现当前宿主不对
-> 重新启动自己
-> 在新宿主里继续
```

但只要存在“重新启动自己”，就必须防止无限循环。

本项目的启动阶段也会发生类似动作：

- 32 位 PowerShell 切到 64 位 PowerShell
- 非稳定入口壳切到更合适的 bootstrap 宿主
- 原始 PowerShell handoff 到管理员 CMD
- payload 残缺后清理并重下

因此本项目必须把重入保护作为启动阶段的一等设计，而不是后期补丁。

#### 重入保护要回答的问题

每次准备重启、重下或 handoff 前，都要能回答：

```text
这件事是否已经试过？
这件事最多允许试几次？
上一次为什么失败？
现在还能不能继续？
如果不能继续，应该在哪里退出？
```

#### 两层重入状态

本项目需要两层重入状态。

第一层是前置重入标记：

```text
用于本地工作区创建之前
用于宿主归一化之前或过程中
```

可通过：

- 命令行参数
- 环境变量
- 临时 marker

传递。

它主要记录：

```text
hostNormalizeAttempted
bitnessNormalizeAttempted
```

第二层是正式启动状态文件：

```text
%LOCALAPPDATA%\dingjiai-installer\state\startup-<startupId>.json
```

它在本地工作区建立后使用，记录完整启动状态。

#### 建议的重入字段

正式状态文件至少应包含：

```text
startupId
stage
hostNormalizeAttempted
bitnessNormalizeAttempted
payloadRepairCount
handoffAttempted
handoffAccepted
lastFailureStage
lastFailureCode
lastFailureMessage
```

这些字段只服务于启动阶段。

#### 建议的重入上限

建议默认上限：

```text
宿主归一化：最多 1 次
位数收敛：最多 1 次
payload 修复 / 重下：v1 固定为 1 次
管理员 CMD handoff：最多 1 次
```

超过上限后应失败退出。

#### 典型保护规则

```text
如果 hostNormalizeAttempted = true，重入后宿主仍不合格：
    启动失败

如果 bitnessNormalizeAttempted = true，重入后仍是错误位数：
    启动失败

如果 payloadRepairCount 达到上限，payload 仍不可信：
    启动失败

如果 handoffAttempted = true，仍未看到 handoffAccepted：
    启动失败或未知状态退出

如果 handoffAccepted = true：
    原始窗口不得继续进入正式菜单
```

#### 重入保护与日志

每一次重入动作都应写日志。

至少记录：

- 准备做什么收敛动作
- 为什么要做
- 当前尝试次数
- 目标宿主或目标阶段
- 成功或失败结果

这能避免用户只看到“闪退”或“不知道发生了什么”。

#### 原始窗口的最终行为

重入保护还要约束原始窗口。

原始 PowerShell 的职责是 bootstrap，不是主 UI。

因此：

```text
看到 handoffAccepted = true
-> 原始窗口退出或显示极简提示

看到 handoff 失败 / 超时
-> 原始窗口显示失败原因和日志位置
-> 不显示正式菜单
```

#### 本项目和 MAS 的差异

MAS 的重入保护主要围绕脚本自身重新进入。

本项目的重入保护既要覆盖脚本自身重新进入，也要覆盖：

```text
payload 状态
管理员 CMD 接管状态
原始窗口与新窗口的职责切换
```

所以本项目要复现 MAS 的保护原则，并扩展到 payload + handoff 架构。

### C. 本地工作区

本地工作区可以固定。

当前建议固定为：

```text
%LOCALAPPDATA%\dingjiai-installer
```

#### 建议包含的子目录

- `cache`
- `payload`
- `logs`
- `state`
- `temp`

#### 需要检查和准备的内容

1. 根目录是否存在，不存在就创建。
2. 子目录是否存在，不存在就创建。
3. 是否已有旧 payload。
4. 旧 payload 是否完整。
5. 是否存在未完成启动残留。
6. 是否已有其它安装器会话占用。

#### 本地工作区与 payload 的职责边界

本地工作区阶段只准备运行基座，不判定 payload 内容可信。

C 阶段负责：

```text
创建目录结构
确认目录可写
建立本次启动的 state / logs / temp 基座
检查并处理会话锁
识别明显残缺或陈旧的临时目录
为 payload 阶段提供干净目标位置
```

D 阶段负责：

```text
下载或读取 manifest
校验 manifest 结构和信任边界
下载 payload 文件
校验 hash / version / required 文件
决定是否清理重下
决定是否允许 last-known-good 候选路径
形成可 handoff 结论
```

也就是说，C 阶段可以发现“payload 目录存在异常”，但不能仅凭目录存在就认为 payload 可用；payload 是否可信、完整、版本一致，只能由 D 阶段给出结论。



#### 这一段的回环

```text
发现旧 payload 目录异常或临时文件残留
-> 不在 C 阶段宣布 payload 可信
-> 准备干净目标位置
-> 交给 D 阶段重新校验或重下
```

这里同样遵循 MAS 式原则：

> **不在脏状态下继续跑，而是先把本地运行基座整理干净。**

### D. payload 获取与校验

这是本项目比 MAS 更需要重点设计的部分。

MAS 的远程入口最终会进入它自己的脚本体系；本项目的远程入口则需要先把本地主程序落地，再交给管理员 CMD 主窗口。

因此，本项目在 MAS 的“入口后移、主逻辑不塞远程入口”的原则上，增加了一个明确的 payload 层。

目标不是“随便下载点文件”，而是：

> **先拿到可信、完整、版本一致、可 handoff 的本地主程序，再 handoff。**

#### payload 层在启动阶段中的角色

payload 层是远程极薄入口和本地管理员 CMD 主程序之间的桥。

它负责：

```text
远程入口
-> 获取 manifest
-> 下载本地主程序 payload
-> 校验完整性
-> 准备 handoff 入口
-> 交给管理员 CMD
```

payload 层不负责安装 Git、Claude 或其它业务组件。

#### 需要检查的内容

1. 分发基地址是否可访问。
2. manifest 是否可下载。
3. manifest 是否可解析。
4. manifest 声明的 payload version 是否明确。
5. manifest 声明的文件清单是否完整。
6. payload 文件是否都能下载。
7. 文件完整性是否匹配。
8. 本地 payload 是否版本一致。
9. 本地主入口是否存在。
10. 主菜单入口是否存在。

#### 关于 manifest + hash 的意思

这里的意思是：

- 不能只看“下载成功了没有”。
- 还要确认“下载下来的这套文件是不是完整且可信”。

做法上可以是：

1. 先下载一个 `manifest.json`。
2. manifest 里声明当前 payload 版本、文件清单、每个文件的 hash。
3. bootstrap 再下载这些文件。
4. 下载后核对 hash。
5. 全部通过，才算 payload 准备完成。

这个机制的目标是避免：

- 文件只下载了一半
- 本地旧缓存和新文件混杂
- 某个文件损坏或不匹配
- CDN / 缓存拿到旧版本内容
- handoff 时入口文件缺失

#### manifest、hash 与真实性边界

`manifest + hash` 只能证明“本地文件与 manifest 声明一致”，不能单独证明“manifest 本身一定来自可信发布者”。

因此文档中的“可信 payload”应拆成两层：

```text
完整性：文件内容与 manifest 中的 hash 一致。
真实性：manifest 或发布元数据来自项目认可的分发源和信任锚。
```

v1 至少应明确：

```text
manifest 只能来自项目定义的分发源。
manifest URL 不接受用户输入拼接。
payload 文件 URL 必须受 manifest 约束，不能跳到任意未知域名。
hash 校验失败时不得 handoff。
```

后续如果引入更强安全边界，应优先考虑：

```text
manifest 签名
固定公钥或证书指纹
发布通道白名单
撤销列表或最小允许版本
```

在没有签名或信任锚之前，文档应避免把 hash 描述成“来源真实性”的充分证明；它首先是完整性校验。

#### payload 版本一致性

payload 不能只按“文件存在”判断。

必须确认：

```text
manifest version
= 本地 payload version
= 本地文件清单对应版本
```

如果本地 payload 中混入不同版本文件，应视为残缺或不可信状态。

处理方式：

```text
清理残缺 payload
-> 重新下载完整 payload
-> 重新校验
```

#### 这一段的回环

建议是**有限重试**，而不是死循环：

```text
下载失败
-> 清理本次残缺文件
-> 重试
-> 再失败
-> 再重试
-> 超过上限则退出
```

建议默认：

```text
payloadRepairCount v1 最多 1 次
```

超过上限后应失败退出。

#### 这一段的范围边界

启动阶段只检查：

> **项目自己的分发源能否提供可信 payload。**

启动阶段的网络硬门槛只应覆盖：

1. 当前宿主是否具备 HTTPS 请求能力。
2. DNS / TLS 是否能支持访问项目分发源。
3. `get.dingjiai.com` 的 manifest 是否可下载。
4. payload 文件是否可下载。
5. 下载后的 payload 是否能通过完整性校验。

外网可达性、GitHub 可达性可以作为诊断信息记录，但不应作为启动阶段硬门槛。

原因是启动阶段的目标不是证明后续所有安装下载都一定成功，而是把可信本地主程序准备好，并成功 handoff 到管理员 `cmd.exe` 主窗口。

不在启动阶段做下面这些事：

- 不测试“整个互联网”
- 不把 GitHub 是否可访问作为启动硬门槛
- 不去巡检 GitHub / npm / winget 等所有外部源
- 不做业务组件的可达性探测

GitHub、winget、npm、GitHub Releases、`raw.githubusercontent.com` 等业务下载源，应当放到安装阶段的 source preflight checkpoint 或各组件 checkpoint 中处理。

当前 `winget` checkpoint 是安装阶段的样板间级 report-only 契约，不属于启动硬门槛。它只做 discovery / diagnosis / decision，输出 `checkpoint.v1` 结构化报告，细分 `healthy`、`missing`、`command_broken`、`command_timeout`、`source_broken`、`source_timeout`、`source_missing`、`source_untrusted`、`helper_failed` 状态，普通报告退出 `0`，helper/runtime failure 退出 `70`。它不得安装、修复、reset/add/update source、修改 PATH、修改注册表或写用户配置；可选结果文件必须限制在用户本地 `dingjiai-installer` root 内。后续 Git、下载器和其它组件 checkpoint 应复用这种“先发现、再诊断、只报告决策、边界可自检”的模型，再分别接入真实动作。

#### payload 与 MAS 的对应关系

MAS 的成熟机制是：

```text
入口薄
真实逻辑后移
进入稳定脚本体系
```

本项目的映射是：

```text
入口薄
真实逻辑后移到本地 payload
进入管理员 CMD 主窗口
```

所以 payload 层不是违背 MAS，而是本项目对 MAS 思路的必要适配。

#### payload 准备完成的标准

payload 准备完成必须同时满足：

```text
manifest 已解析
payload version 明确
所有文件已下载
所有 hash 校验通过
本地主入口存在
主菜单入口存在
handoff 所需参数可以组装
```

只要其中任一项不满足，就不能进入 handoff。

### E. handoff 到管理员 CMD

这一段是启动阶段成功与否的最终判定点。

MAS 的成熟机制强调：

```text
入口宿主不是最终战场
要把流程交给正确权限、正确宿主下的主程序
```

本项目在这个原则上进一步明确：

```text
新的管理员 cmd.exe 主窗口
= Windows 路线下的真实主战场
```

因此 handoff 不是普通的“启动一个窗口”，而是启动阶段的最终交棒动作。

#### 需要检查的内容

1. 本地主入口文件是否存在。
2. 主菜单入口文件是否存在。
3. payload version 是否已经确认可信。
4. handoff 命令是否可正确组装。
5. handoff 参数是否完整。
6. 管理员 `cmd.exe` 是否能启动。
7. 新窗口启动后是否真正进入本地主入口。
8. 新窗口是否写回 `handoffAccepted` 接管标记。

#### handoff 参数契约

handoff 到管理员 CMD 时，至少应传递：

```text
startupId
workspaceRoot
payloadRoot
mainEntryPath
logPath
statePath
source=bootstrap
handoffMode=admin-cmd
```

这些参数用于让新窗口明确知道：

```text
自己接管的是哪一次启动
本地 payload 在哪里
状态文件在哪里
日志写到哪里
应该进入哪个主入口
```

#### 成功标准

不是“提权命令发出去了”，也不是“cmd 窗口打开了”。

唯一成功标准是：

> **管理员 `cmd.exe` 新主窗口已成功启动，并接管进入安装器主入口。**

更具体地说：

```text
新管理员 CMD 进入本地主入口
-> 写回 handoffAccepted = true
-> 原始 PowerShell 看到接管标记
-> 原始 PowerShell 退出或显示极简完成提示
```

#### 新 CMD 写回接管标记

新管理员 CMD 接管后，应写回状态文件：

```json
{
  "stage": "completed",
  "handoffAccepted": true
}
```

这个写回动作是启动阶段的关键确认点。

没有这个标记，就不能假设 handoff 成功。

#### 失败情况

以下情况都算启动失败：

- 提权被拒绝
- 新窗口没起来
- 新窗口起来了但没进入主入口
- 新窗口没有写回接管标记
- 入口文件缺失
- handoff 命令拼装失败
- handoff 参数缺失
- 状态文件无法写回

#### 失败处理

handoff 失败后，原始 PowerShell 只应该做三件事：

```text
显示简短失败原因
显示日志位置
退出
```

它不应该：

```text
降级显示正式菜单
继续执行安装任务
反复弹出 UAC
偷偷改用非管理员模式
```

#### handoff 尝试次数

建议规则：

```text
管理员 CMD handoff：最多 1 次正式尝试
```

因为 UAC 是用户可见动作，多次重复弹出会造成困惑和不信任。

#### 已经是管理员时仍然 handoff

如果用户从管理员 PowerShell 启动入口，仍然建议 handoff 到专用管理员 CMD 主窗口。

原因是：

```text
管理员权限 ≠ 稳定主 UI 宿主
```

本项目的稳定主 UI 宿主已经定义为新的管理员 CMD。

因此原始 PowerShell 即使已经管理员，也仍然只是入口壳。

#### 和 MAS 的对应关系

MAS 的 handoff 思想是：

```text
不要被原始入口壳限制
把脚本交给正确权限 / 正确宿主继续执行
```

本项目的映射是：

```text
不要让原始 PowerShell 成为主程序
把本地 payload 交给管理员 cmd.exe 主窗口继续执行
```

这就是本项目对 MAS 启动机制的核心复现之一。

#### 这一段的边界

- 不在原始 PowerShell 窗口继续显示正式菜单。
- 不在原始窗口继续跑安装任务。
- 不把 handoff 发起成功当作启动成功。
- 不允许无限 handoff。

## 六、启动阶段里的 3 个回环

如果按 MAS 风格来做，启动阶段应该明确有 3 个回环。

### 回环 1：宿主归一化回环

```text
宿主不对
-> 自重启到正确宿主
-> 重新进入启动流程
```

这是 MAS 最经典的套路之一。

### 回环 2：payload 修复回环

```text
本地 payload 残缺 / 远程下载失败
-> 清理残缺文件
-> 重下
-> 再校验
```

这是本项目比 MAS 更需要重点设计的回环。

### 回环 3：handoff 单次重入保护

```text
准备 handoff
-> 启动管理员 CMD
-> 若失败则退出
-> 不允许无限 handoff
```

这里不建议做无限 handoff 重试。handoff 更适合严格受控的一次或有限次数尝试。

### 兼容性收敛的核心规则

兼容性收敛是启动阶段最重要的设计点之一。

启动阶段不应该在错误宿主、错误位数、错误权限或脏 payload 状态下继续硬跑，而应该采用 MAS 风格的收敛方式：

```text
发现当前运行形态不适合继续
-> 切换到更正确的运行形态
-> 带状态标记重新进入启动流程
-> 重新检查
-> 通过后继续
-> 仍不通过则明确失败退出
```

这里的重点不是“兼容所有坏环境”，而是：

> **把可修正的不稳定运行环境，尽快收敛到项目定义的稳定运行环境。**

当前建议把启动阶段的稳定目标定义为：

```text
64 位 Windows
+ 可用 PowerShell bootstrap 宿主
+ 可写本地工作区
+ 完整可信 payload
+ 新开的管理员 cmd.exe 主窗口
```

#### 哪些情况适合收敛

适合用“修正后重入”的方式处理：

1. 32 位 PowerShell 跑在 64 位 Windows 上。
2. 当前宿主能力不足，但系统里存在更合适的 PowerShell 宿主。
3. 当前没有管理员权限，但可以通过 UAC 提权进入管理员 CMD。
4. 本地 payload 残缺，但可以清理后重新下载。
5. 工作区目录缺失，但可以创建。
6. 上一次启动残留临时文件，但可以安全清理。

这些属于“环境形态不理想，但可被安装器修正”的问题。

#### 哪些情况不适合收敛

不适合继续绕路，应该直接失败退出：

1. 不是 Windows。
2. Windows 版本低于最低支持范围。
3. 系统架构不在支持范围内。
4. `cmd.exe` 不可用。
5. `Start-Process -Verb RunAs` 不可用。
6. 用户拒绝 UAC 提权。
7. 本地工作区不可写且无法修复。
8. payload 下载或校验超过重试上限仍失败。
9. handoff 已尝试过但仍无法被新管理员 CMD 接管。

这些属于“继续往后走一定会失败，或者不能由安装器安全修正”的问题。

#### 收敛必须带状态标记

每一次收敛都必须带状态标记，不能只靠当前进程记忆。

启动阶段至少需要记录：

- `startupId`：本次启动会话 ID
- `stage`：当前启动阶段
- `hostNormalizeAttempted`：是否已经尝试过宿主归一化
- `payloadRepairCount`：payload 修复 / 重下次数
- `handoffAttempted`：是否已经尝试过管理员 CMD handoff
- `lastFailureCode`：上一次失败原因
- `lastFailureStage`：上一次失败阶段

这些状态建议写入本地工作区的 `state` 或 `temp` 下，而不是只放在内存变量里。

#### 收敛次数必须有限

启动阶段所有回环都必须有限。

建议规则：

```text
宿主归一化：最多 1 次修正重入
payload 修复：v1 最多 1 次有限重试
管理员 CMD handoff：最多 1 次正式尝试
```

超过上限后应明确失败，并把失败阶段和建议动作写入日志。

#### 原始窗口和新窗口的职责边界

收敛完成后，原始 PowerShell / Terminal 仍然只是 bootstrap 壳。

一旦管理员 `cmd.exe` 新窗口成功接管：

```text
原始窗口：退出或显示极简完成提示
新管理员 CMD：进入本地主入口和主菜单
```

如果 handoff 失败：

```text
原始窗口：显示启动失败原因和日志位置
原始窗口：不得降级显示正式主菜单
```

也就是说，兼容性收敛只能把用户带到正确主战场，不能把原始窗口变成备用主程序。

### 启动状态文件与重入标记

兼容性收敛不能只依赖当前进程里的变量。

原因是启动阶段可能发生：

- 从 32 位 PowerShell 重启到 64 位 PowerShell
- 从原始 PowerShell 切换到管理员 `cmd.exe`
- payload 清理后重新下载
- 原始窗口退出，新窗口继续接管

这些动作都会跨进程，甚至跨窗口。

因此，启动阶段需要一个明确的本地状态文件，用来记录本次启动已经走到哪里、做过哪些收敛动作、是否还能继续重试。

建议位置：

```text
%LOCALAPPDATA%\dingjiai-installer\state\startup-<startupId>.json
```

其中 `startupId` 是本次一行命令启动时生成的唯一会话 ID。

#### 状态文件建议字段

建议至少包含：

```json
{
  "startupId": "...",
  "createdAt": "...",
  "updatedAt": "...",
  "stage": "host-normalize | platform-gate | workspace | payload | handoff | completed | failed",
  "entryShell": "powershell",
  "is64BitProcess": true,
  "isAdmin": false,
  "workspaceRoot": "%LOCALAPPDATA%\\dingjiai-installer",
  "payloadVersion": "...",
  "hostNormalizeAttempted": false,
  "payloadRepairCount": 0,
  "handoffAttempted": false,
  "handoffAccepted": false,
  "lastFailureStage": null,
  "lastFailureCode": null,
  "lastFailureMessage": null
}
```

这些字段不是业务状态，而是启动阶段自己的运行状态。

#### stage 的含义

`stage` 应该只描述启动阶段走到哪一步：

```text
host-normalize  宿主归一化中
platform-gate   平台硬门槛检查中
workspace       本地工作区准备中
payload         payload 获取与校验中
handoff         管理员 CMD handoff 中
completed       新管理员 CMD 已接管
failed          启动失败
```

这样后续排查问题时，可以明确知道启动失败发生在哪一段。

#### 重入标记的作用

重入标记用于回答三个问题：

```text
这件事有没有试过？
还能不能再试？
如果不能再试，应该在哪里失败退出？
```

建议规则：

1. `hostNormalizeAttempted = true` 后，如果重新进入仍然宿主不正确，就失败退出。
2. `payloadRepairCount` 达到上限后，不再清理重下，直接失败退出。
3. `handoffAttempted = true` 后，不再无限启动管理员窗口。
4. `handoffAccepted = true` 后，原始窗口不得继续进入主菜单。

#### 新 CMD 接管确认

handoff 不能只看“管理员窗口是否被启动”。

建议由新管理员 `cmd.exe` 进入本地主入口后，写回状态：

```json
{
  "stage": "completed",
  "handoffAccepted": true
}
```

原始 PowerShell 只在看到这个接管标记后，才把启动阶段视为成功。

如果新窗口启动了但没有写回接管标记，应视为 handoff 失败或未知，不应在原始窗口继续显示正式菜单。

#### 失败记录

启动失败时，应写入：

- `stage = failed`
- `lastFailureStage`
- `lastFailureCode`
- `lastFailureMessage`

失败信息要能支持用户看到类似结果：

```text
启动失败：无法启动管理员 CMD 主窗口。
建议：请确认没有拒绝 UAC 提权，然后重新运行一行命令。
日志位置：%LOCALAPPDATA%\dingjiai-installer\logs\...
```

#### 状态文件生命周期

状态文件不应永久无限堆积。

建议规则：

1. 本次启动成功后，可以保留最近一次启动状态用于诊断。
2. 旧的启动状态可以按时间清理。
3. 正在运行的启动状态不应被另一个启动流程随意覆盖。
4. 如果发现很久以前的未完成状态，应视为僵尸状态，允许安全清理。

#### 和业务状态分离

启动状态文件只服务于启动阶段。

它不应该记录：

- Git 是否已安装
- Claude 是否已安装
- gh / rg / Python 等业务组件状态
- 用户 API key
- 用户配置内容

这些属于安装阶段、组件 checkpoint 或产品配置阶段，不属于启动状态。

## 十、MAS 启动机制复现总表

当前已确认：启动阶段要尽量复现 MAS 的成熟机制，而不是重新发明一套启动系统。

这里的“复现”指的是复现机制、流程和边界，不是复制 MAS 的 GPL-3.0 代码。

### 1. 总体映射

| MAS 成熟机制 | MAS 中的作用 | 本项目中的映射 | 本项目边界 |
| --- | --- | --- | --- |
| 极薄远程入口 | 降低用户进入成本 | `irm https://get.dingjiai.com/win.ps1 \| iex` 只负责 bootstrap | 不显示正式菜单，不做业务安装 |
| 宿主归一化 | 不在错误宿主里硬跑 | PowerShell 入口只做临时宿主，必要时切到正确宿主 | 不把原始 shell 当长期 UI |
| 位数收敛 | 64 位系统上避免 32 位进程陷阱 | 64 位 Windows 上尽早收敛到 64 位 PowerShell 语义 | 不处理业务组件架构 |
| 权限收敛 | 先解决权限再进主功能 | 通过 UAC handoff 到管理员 `cmd.exe` 主窗口 | 不在非管理员原窗口继续正式流程 |
| 重入保护 | 防止自重启无限循环 | 前置重入标记 + `startup-<startupId>.json` | 所有回环必须有限 |
| 早失败平台门槛 | 不支持就早点停 | Windows 版本、架构、cmd、RunAs、写入能力等硬门槛 | 不检查 GitHub / npm / winget 业务源 |
| 主逻辑后移 | 入口不承载全部业务 | 下载并校验本地 payload 后再进入主程序 | payload 只保证本项目主程序可信 |
| 稳定宿主接管 | 进入可控主流程 | 新管理员 CMD 写回 `handoffAccepted` | `Start-Process` 成功不等于接管成功 |

### 2. 本项目启动阶段最终链路

```text
用户执行一行命令
-> 极薄 PowerShell 入口
-> 宿主归一化
-> 位数收敛
-> 平台硬门槛早失败
-> 准备本地工作区
-> 获取 manifest
-> 下载 payload
-> 校验 hash / version / 文件清单
-> 准备 handoff 参数
-> UAC 启动管理员 cmd.exe
-> 新 CMD 进入本地主入口
-> 新 CMD 写回 handoffAccepted
-> 原始 PowerShell 退出
-> 管理员 CMD 显示主菜单
```

### 3. 启动阶段必须坚持的边界

启动阶段只解决：

```text
能不能把可信本地主程序交给管理员 CMD 主窗口接管
```

启动阶段不解决：

```text
Git 是否存在
Claude 是否存在
gh / rg / Python 是否存在
GitHub 是否可访问
npm / winget 是否健康
PATH 是否需要修复
用户 settings.json 是否完整
```

这些后移到安装阶段的 source preflight checkpoint、组件 checkpoint 或产品配置阶段。

### 4. 启动阶段的定稿原则

```text
启动阶段严格参考 MAS 的成熟启动机制；
除非 MAS 的做法和本项目“payload + 管理员 CMD 主窗口”目标冲突，
否则不自行发明新机制。
```

实现要求：

```text
学习机制
重新实现
不复制代码
保持 MIT 兼容
```

### 5. 后续文档细化顺序

后续如果继续拆启动阶段，应优先围绕这些专题继续写文档：

1. MAS 风格入口宿主检查字段清单。
2. MAS 风格 32 位到 64 位重启细节。
3. MAS 风格管理员权限判断与 UAC handoff 细节。
4. 前置重入标记与正式 startup 状态文件字段。
5. payload manifest 字段、hash 规则与版本一致性。
6. `handoffAccepted` 写回机制与超时判断。
7. last-known-good payload 是否允许作为远程失败时的容灾路径。

## 十一、字段级深挖：入口宿主检查字段

本节把 MAS 风格的“先判断宿主是否合格，再决定是否收敛”落到字段级。

入口宿主检查只回答一个问题：

```text
当前原始 PowerShell 是否足够完成 bootstrap，或者是否需要先收敛到更合适的宿主？
```

它不回答业务安装问题。

### 1. 字段总表

| 字段 | 检查目的 | 失败处理 | 是否硬门槛 |
| --- | --- | --- | --- |
| `isWindows` | 确认当前系统是 Windows | 非 Windows 直接失败 | 是 |
| `psAvailable` | 确认当前在 PowerShell 语义中运行 | 不能解释 PowerShell 脚本则失败 | 是 |
| `psVersion` | 确认 PowerShell 满足 bootstrap 最低能力 | 低于最低版本则失败或提示换宿主 | 是 |
| `osArchitecture` | 确认 OS 架构 | 不支持架构则失败 | 是 |
| `processArchitecture` | 确认当前进程位数 | 可收敛则重启到正确位数；不可收敛则失败 | 是 |
| `canResolveSystemPowerShell` | 能否定位系统内置 PowerShell | 无法定位且当前宿主不合格则失败 | 是 |
| `canUseHttps` | 能否发起 HTTPS 请求 | 无法访问项目分发源则失败 | 是 |
| `canWriteTemp` | 能否写临时文件 | 临时写入不可用则失败 | 是 |
| `canStartProcess` | 能否启动新进程 | 不能启动新进程则失败 | 是 |
| `canFindCmd` | 能否定位 `cmd.exe` | `cmd.exe` 不可用则失败 | 是 |
| `canUseRunAs` | 能否发起 UAC / RunAs | 不可用则失败 | 是 |
| `hostNormalizeAttempted` | 是否已经尝试宿主归一化 | 已尝试仍失败则退出 | 防循环字段 |
| `bitnessNormalizeAttempted` | 是否已经尝试位数收敛 | 已尝试仍失败则退出 | 防循环字段 |

### 2. 字段边界

这些字段只覆盖启动宿主能力。

不加入：

- Git 是否存在。
- Claude 是否存在。
- winget 是否存在。
- GitHub 是否可访问。
- npm 是否可访问。
- 用户配置是否存在。
- PATH 是否正确。

### 3. 建议检查顺序

入口宿主检查建议顺序：

```text
isWindows
-> psAvailable
-> psVersion
-> osArchitecture
-> processArchitecture
-> canResolveSystemPowerShell
-> canWriteTemp
-> canStartProcess
-> canFindCmd
-> canUseRunAs
-> canUseHttps
```

原因：

1. 先排除完全不支持的平台。
2. 再排除脚本无法继续解释的宿主。
3. 再处理位数和宿主收敛。
4. 再检查后续 bootstrap 必需能力。
5. HTTPS 放在后面，因为它依赖当前宿主已经基本可用。

### 4. 检查结果分类

检查结果建议分三类：

```text
pass       当前字段通过
converge   当前字段不合格，但可通过收敛修正
fail       当前字段不合格，且不可安全修正
```

例如：

```text
64 位 OS + 32 位进程 + 未尝试过位数收敛
= converge

64 位 OS + 32 位进程 + 已尝试过位数收敛
= fail

cmd.exe 不存在
= fail
```

### 5. 输出原则

入口宿主检查失败时，用户看到的提示应该简单：

```text
启动失败：当前 PowerShell 环境不支持继续启动。
建议：请打开系统自带 PowerShell 后重新运行一行命令。
```

日志里再记录具体字段：

```text
stage=host-normalize
field=processArchitecture
expected=x64
actual=x86
attempted=true
result=fail
```

## 十二、字段级深挖：32 位到 64 位重启细节

本节把 MAS 风格的“位数收敛”继续拆到执行边界。

它要解决的典型问题是：

```text
64 位 Windows 上，用户从 32 位 PowerShell / 32 位宿主进入 bootstrap。
```

这种情况下，如果继续在 32 位进程语义里跑，后续可能遇到系统路径重定向、注册表视图差异、工具解析不一致等问题。

因此，本项目启动阶段应当尽早收敛到 64 位 PowerShell 进程，再继续平台门槛、本地工作区、payload 和 handoff。

### 1. 触发条件

位数收敛只在以下条件同时满足时触发：

```text
isWindows = true
osArchitecture = x64
processArchitecture = x86
bitnessNormalizeAttempted = false
canResolveSystemPowerShell = true
```

含义是：

```text
系统本身支持 64 位，当前进程却是 32 位，并且还有一次安全重启到 64 位宿主的机会。
```

如果 OS 本身不是 64 位，则不是“位数收敛”问题，而是平台支持问题，应进入平台硬门槛判断。

### 2. 收敛目标

目标宿主应优先使用系统内置 64 位 Windows PowerShell。

概念上目标是：

```text
64 位 Windows PowerShell
+ 相同 bootstrap 入口
+ 相同 startupId 或前置标记
+ 明确 bitnessNormalizeAttempted = true
```

这里不应切到不确定的用户自装 shell，也不应依赖用户当前 Terminal profile。

原因是启动阶段追求稳定和可预测，不追求跟随用户偏好的 shell。

### 3. 前置标记

位数收敛发生得很早，可能早于正式工作区和 `startup-<startupId>.json` 创建。

因此必须有前置标记记录：

```text
bitnessNormalizeAttempted = true
```

这个标记的作用不是记录完整启动状态，而是防止无限重启。

它至少要回答：

```text
当前启动链路是否已经尝试过从 32 位进程切到 64 位进程？
```

### 4. 重启后的复查

重启到目标宿主后，不能假设已经成功。

必须重新检查：

```text
isWindows
psAvailable
psVersion
osArchitecture
processArchitecture
bitnessNormalizeAttempted
canWriteTemp
canStartProcess
canFindCmd
canUseRunAs
canUseHttps
```

其中最关键的是：

```text
processArchitecture 是否已经从 x86 变为 x64
```

只有复查通过，才允许继续进入后续启动阶段。

### 5. 失败条件

以下情况应明确失败，而不是继续尝试：

```text
64 位 OS + 当前仍是 32 位进程 + bitnessNormalizeAttempted = true
```

也就是说：

```text
已经尝试过一次位数收敛，但重入后仍未进入 64 位进程。
```

此时继续重启没有意义，必须停止，避免无限循环。

用户提示应保持简单：

```text
启动失败：无法切换到 64 位 PowerShell 环境。
建议：请打开系统自带 64 位 PowerShell 后重新运行一行命令。
```

日志中记录具体原因：

```text
stage=host-normalize
field=processArchitecture
expected=x64
actual=x86
bitnessNormalizeAttempted=true
result=fail
```

### 6. 不在这里处理的内容

位数收敛只处理 bootstrap 宿主自身的进程位数。

不处理：

- Git 是 32 位还是 64 位。
- Claude 安装包架构。
- winget 包架构。
- Node / Python / gh / rg 的架构。
- PATH 中哪个架构的工具优先。
- 注册表里的业务组件探测。

这些属于后续组件 checkpoint 的 discovery / allowance 范围。

### 7. 位数收敛在启动链路中的位置

位数收敛应发生在平台硬门槛之前，或作为平台硬门槛的前置收敛动作。

推荐顺序是：

```text
入口开始
-> 判断 Windows / PowerShell
-> 判断 OS 架构
-> 判断当前进程位数
-> 如需收敛，重启到 64 位 PowerShell
-> 重入后复查
-> 进入平台硬门槛
```

这样可以保证后面的工作区、payload、handoff 都在统一的 64 位进程语义下执行。

## 十三、字段级深挖：管理员权限判断与 UAC handoff 细节

本节把 MAS 风格的“先解决权限，再进入主逻辑”映射到本项目的 Windows 启动阶段。

本项目和 MAS 的差异在于：

```text
MAS 提权后继续进入自己的业务菜单；
本项目提权后必须进入新的管理员 cmd.exe 主窗口，并由该窗口接管本地主入口。
```

因此，UAC handoff 不是单纯“拿到管理员权限”，而是“拿到管理员权限 + 切到稳定 CMD 主宿主 + 接管 payload 主入口”。

### 1. 权限判断的目标

权限判断只回答一个问题：

```text
当前是否已经具备进入真实主 UI 的管理员执行条件？
```

但它不能把“当前进程是管理员”误判为“启动阶段成功”。

原因是：

```text
管理员权限是必要条件；
专用管理员 cmd.exe 主窗口接管才是启动阶段成功条件。
```

也就是说，即使原始 PowerShell 已经是管理员，仍然不应该直接在原始窗口显示正式菜单。

### 2. 字段总表

| 字段 | 检查目的 | 失败处理 | 是否硬门槛 |
| --- | --- | --- | --- |
| `isAdmin` | 当前进程是否已经管理员 | 不是管理员则准备 UAC handoff | 不是单独成功条件 |
| `canUseRunAs` | 是否能发起 UAC 提权 | 不能发起则失败 | 是 |
| `canFindCmd` | 是否能定位 `cmd.exe` | 找不到则失败 | 是 |
| `mainEntryPath` | 本地主入口是否存在 | 不存在则回到 payload 阶段修复或失败 | 是 |
| `payloadRoot` | 本地 payload 根目录是否明确 | 不明确则失败 | 是 |
| `statePath` | 启动状态文件路径是否明确 | 不明确则失败 | 是 |
| `logPath` | handoff 后日志路径是否明确 | 不明确则失败 | 是 |
| `handoffAttempted` | 是否已经尝试 handoff | 已尝试仍失败则退出 | 防循环字段 |
| `handoffAccepted` | 新 CMD 是否写回接管标记 | 未写回则不算成功 | 成功判定字段 |

### 3. UAC handoff 触发条件

进入 handoff 前，必须满足：

```text
host normalize 已完成
bitness normalize 已完成或无需执行
platform gate 已通过
workspace 已准备
manifest 已解析
payload 已下载并校验通过
mainEntryPath 已存在
canFindCmd = true
canUseRunAs = true
handoffAttempted = false
```

只要还没有满足这些条件，就不应该发起管理员 CMD。

特别是：

```text
payload 未完整校验通过，不允许 handoff。
```

因为管理员 CMD 一旦打开，就会成为真实主战场，不能让它接管半截 payload。

### 4. handoff 目标

handoff 的目标不是“打开任意管理员窗口”，而是：

```text
新打开一个管理员 cmd.exe
-> 进入本地 payload 主入口
-> 带上启动参数
-> 写回接管状态
-> 显示正式主菜单
```

稳定目标是：

```text
administrator cmd.exe
+ local payload main entry
+ startup state path
+ log path
+ handoffAccepted write-back
```

### 5. 参数契约

handoff 参数至少应包含：

```text
startupId
workspaceRoot
payloadRoot
mainEntryPath
statePath
logPath
source=bootstrap
handoffMode=admin-cmd
```

这些参数的用途是：

| 参数 | 作用 |
| --- | --- |
| `startupId` | 把原始 bootstrap 与新 CMD 接管动作关联起来 |
| `workspaceRoot` | 告诉新 CMD 本地工作区位置 |
| `payloadRoot` | 告诉新 CMD 本地主程序位置 |
| `mainEntryPath` | 明确真正进入的主入口 |
| `statePath` | 让新 CMD 写回状态 |
| `logPath` | 让新 CMD 继续写同一启动日志 |
| `source=bootstrap` | 标记来源是远程入口引导 |
| `handoffMode=admin-cmd` | 标记当前接管模式 |

### 6. handoff 参数与路径安全边界

handoff 参数会被传入管理员 `cmd.exe`，因此必须把它当作安全边界处理。

基本规则：

```text
所有路径必须先规范化为绝对路径。
所有路径必须落在项目允许的 workspaceRoot / payloadRoot / logs / state 范围内。
mainEntryPath 必须来自已校验 manifest 的 required 文件，不能来自用户输入。
参数名必须白名单化，未知参数不得透传到主入口。
参数值必须按 cmd.exe 规则安全转义。
不得通过字符串拼接执行未验证命令。
不得依赖当前目录、PATH 搜索或环境变量展开来定位主入口。
```

还应拒绝以下路径形态：

```text
指向 workspaceRoot 外部的相对路径
包含 .. 跳出目录的路径
可疑重解析点 / junction / symlink 跳转
UNC 或网络路径作为主入口
用户可写但未纳入 manifest 校验的入口文件
```

handoff 的目标必须是“已校验 payload 中的主入口”，而不是“某个刚好叫 main 的文件”。



### 7. handoffAttempted 写入时机

`handoffAttempted` 应在发起 UAC handoff 之前写入。

原因是：

```text
UAC 弹窗出现后，原始进程可能失去连续控制；
如果等 Start-Process 返回后再写，失败路径可能漏记。
```

推荐顺序：

```text
确认 payload 可 handoff
-> 写入 handoffAttempted = true
-> 写入 stage = handoff
-> 发起 Start-Process -Verb RunAs
-> 等待 handoffAccepted 或超时
```

这样即使提权失败、用户取消 UAC、cmd.exe 启动失败，也不会进入无限 handoff。

### 8. 用户取消 UAC

用户取消 UAC 不应被包装成复杂错误。

用户提示可以是：

```text
启动已取消：需要管理员权限才能打开安装器主窗口。
```

日志中记录：

```text
stage=handoff
field=canUseRunAs
handoffAttempted=true
handoffAccepted=false
result=cancelled-or-failed
```

启动阶段应停止，不应退回原始 PowerShell 显示正式菜单。

### 9. 原始 PowerShell 的退出条件

原始 PowerShell 发起 handoff 后，只做两件事：

```text
等待新 CMD 写回 handoffAccepted
或等待超时 / 失败
```

如果看到：

```json
{
  "stage": "completed",
  "handoffAccepted": true
}
```

则原始 PowerShell 可以退出，或显示极简提示后退出。

如果未看到该标记，则启动阶段不算成功。

### 10. 不允许的降级路径

UAC handoff 失败后，不应降级为：

- 在原始 PowerShell 中显示正式菜单。
- 在非管理员窗口继续安装。
- 打开非管理员 `cmd.exe` 作为主 UI。
- 反复弹 UAC。
- 让用户手动复制内部 payload 命令。

这些都会破坏“一个稳定管理员 CMD 主战场”的架构约束。

### 11. 成功标准

handoff 的唯一成功标准是：

```text
新管理员 cmd.exe 已进入本地主入口，且写回 handoffAccepted = true。
```

不是：

```text
Start-Process 调用成功
cmd.exe 进程出现
UAC 弹窗被用户点了是
原始 PowerShell 没报错
```

这些最多只能说明 handoff 已发起，不能说明新主窗口已经接管。

## 十四、字段级深挖：启动状态字段

本节把启动阶段的状态文件拆成字段级契约。

启动状态文件的目的不是记录整个安装器状态，而是只记录“一次启动链路”是否已经安全完成。

推荐路径是：

```text
%LOCALAPPDATA%\dingjiai-installer\state\startup-<startupId>.json
```

它只服务启动阶段，不记录业务组件安装状态。

### 1. 状态文件的职责

启动状态文件只回答这些问题：

```text
这次启动现在走到哪一段？
哪些收敛动作已经尝试过？
payload 是否已经准备到可 handoff 状态？
是否已经尝试 UAC handoff？
新管理员 CMD 是否已经接管？
如果失败，失败在哪一段、为什么失败？
```

它不回答：

```text
Git 是否安装成功
Claude 是否安装成功
gh / rg / Python 是否健康
用户 API key 是否填写
settings.json 是否写好
```

这些都属于后续安装阶段或产品配置阶段。

### 2. 字段总表

| 字段 | 类型 | 作用 |
| --- | --- | --- |
| `startupId` | string | 当前启动会话 ID |
| `createdAt` | string | 启动状态创建时间 |
| `updatedAt` | string | 最近一次状态更新时间 |
| `stage` | string | 当前启动阶段 |
| `entryShell` | string | 原始入口宿主摘要 |
| `is64BitProcess` | boolean | 当前 bootstrap 是否在 64 位进程语义中 |
| `isAdmin` | boolean | 当前记录状态时是否管理员 |
| `workspaceRoot` | string | 本地工作区根目录 |
| `payloadRoot` | string | 本地 payload 根目录 |
| `payloadVersion` | string | 当前准备 handoff 的 payload 版本 |
| `manifestUrl` | string | 本次使用的 manifest 地址 |
| `manifestHash` | string | manifest 自身摘要或记录用 hash |
| `hostNormalizeAttempted` | boolean | 是否尝试过宿主归一化 |
| `bitnessNormalizeAttempted` | boolean | 是否尝试过位数收敛 |
| `payloadRepairCount` | number | payload 清理 / 重下次数 |
| `handoffAttempted` | boolean | 是否已经尝试管理员 CMD handoff |
| `handoffAccepted` | boolean | 新管理员 CMD 是否写回接管成功 |
| `lastFailureStage` | string | 最近失败阶段 |
| `lastFailureCode` | string | 最近失败代码 |
| `lastFailureMessage` | string | 面向日志的失败说明 |

字段可以随实现微调，但必须保持边界：

```text
只记录启动阶段状态，不记录业务组件状态。
```

### 3. stage 枚举

`stage` 建议使用有限枚举：

```text
host-normalize
platform-gate
workspace
payload
handoff
completed
failed
```

含义：

| stage | 含义 |
| --- | --- |
| `host-normalize` | 正在处理宿主、PowerShell、位数等收敛 |
| `platform-gate` | 正在检查不可继续的硬门槛 |
| `workspace` | 正在准备本地工作区 |
| `payload` | 正在获取、修复、校验 payload |
| `handoff` | 正在交给管理员 CMD 主窗口 |
| `completed` | 新管理员 CMD 已接管 |
| `failed` | 启动阶段明确失败 |

不要为业务组件增加 stage，例如：

```text
git-install
claude-install
winget-check
settings-write
```

这些不属于启动状态文件。

### 4. 状态更新原则

状态更新应遵循：

```text
进入一个阶段前，先写 stage。
执行一次会改变回环次数的动作前，先写 attempted / count。
失败时，写 failed + failure 字段。
handoff 成功只能由新管理员 CMD 写回。
```

尤其是：

```text
handoffAccepted = true
```

不应该由原始 PowerShell 自己写。

原始 PowerShell 只能读取它，不能伪造它。

### 5. 回环字段规则

受控回环必须有显式字段。

建议规则：

| 回环 | 字段 | 上限 |
| --- | --- | --- |
| 宿主归一化 | `hostNormalizeAttempted` | 1 次 |
| 位数收敛 | `bitnessNormalizeAttempted` | 1 次 |
| payload 修复 / 重下 | `payloadRepairCount` | v1 固定 1 次 |
| 管理员 CMD handoff | `handoffAttempted` | 1 次 |

当达到上限仍失败时，必须进入：

```text
stage = failed
```

而不是继续重试。

### 6. 失败字段规则

失败时至少记录：

```json
{
  "stage": "failed",
  "lastFailureStage": "payload",
  "lastFailureCode": "payload-hash-mismatch",
  "lastFailureMessage": "Payload hash verification failed."
}
```

用户提示可以简单，但日志要可定位。

推荐方式是：

```text
用户提示：一句中文说明 + 下一步建议
日志字段：阶段、字段、期望值、实际值、是否重试过、最终结果
```

### 7. 与前置重入标记的关系

正式状态文件依赖本地工作区。

但宿主归一化、位数收敛可能发生在工作区创建前。

因此要区分：

```text
前置重入标记：用于工作区创建前的最早期防循环。
正式启动状态文件：用于工作区建立后的完整启动状态。
```

一旦工作区创建成功，应把可继承的前置标记写入正式状态文件，例如：

```text
hostNormalizeAttempted
bitnessNormalizeAttempted
```

这样后续阶段只读正式状态文件即可。

### 8. 状态文件不保存敏感信息

启动状态文件不得保存：

- API key。
- 用户输入的服务 URL。
- Claude 配置内容。
- 私有 token。
- GitHub 登录状态。
- npm token。

启动阶段只需要记录启动链路本身，不需要知道用户产品配置。

## 十五、字段级深挖：manifest 字段、hash 规则与版本一致性

本节把 payload 获取与校验拆到 manifest 字段级。

启动阶段拉 manifest 的目的不是“看看有没有新版本”，而是确认：

```text
远程声明的 payload 是什么版本、包含哪些文件、每个文件应当是什么 hash、哪些文件构成可 handoff 的完整本地主程序。
```

只有 manifest 可解析、payload 文件齐全、hash 校验通过、入口文件存在，才允许进入管理员 CMD handoff。

### 1. manifest 的职责

manifest 只服务启动 payload。

它回答：

```text
这次应该使用哪一版 payload？
这一版 payload 应包含哪些文件？
每个文件下载地址是什么？
每个文件的 hash 是什么？
哪个文件是主入口？
哪个文件是菜单入口？
是否满足最低 bootstrap 协议？
```

它不回答：

```text
Git 应该装哪个版本
Claude 应该装哪个版本
winget 是否健康
GitHub 是否可访问
用户 settings.json 应该写什么
```

### 2. manifest 建议字段

manifest 建议至少包含：

| 字段 | 作用 |
| --- | --- |
| `schemaVersion` | manifest 结构版本 |
| `payloadVersion` | payload 版本 |
| `generatedAt` | manifest 生成时间 |
| `minBootstrapVersion` | 最低 bootstrap 协议版本 |
| `platform` | 目标平台，例如 `windows` |
| `architecture` | 目标架构，例如 `x64` |
| `files` | payload 文件清单 |
| `mainEntry` | handoff 后进入的主入口 |
| `menuEntry` | 主菜单入口或菜单定义文件 |
| `hashAlgorithm` | hash 算法，例如 `sha256` |
| `releaseChannel` | 发布通道，例如 `stable` |

`files` 中每个文件建议包含：

| 字段 | 作用 |
| --- | --- |
| `path` | 文件在 payloadRoot 下的相对路径 |
| `url` | 下载地址 |
| `size` | 文件大小，用于基础完整性判断 |
| `sha256` | 文件内容 hash |
| `required` | 是否为 handoff 必需文件 |

### 3. hash 规则

hash 校验应按文件逐个执行。

基本规则：

```text
下载文件
-> 计算本地 hash
-> 与 manifest 中声明的 hash 对比
-> 全部 required 文件通过
-> 才能继续
```

不能只判断：

```text
HTTP 下载成功
文件存在
文件大小大于 0
```

这些都不足以证明 payload 完整可信。

### 4. 版本一致性规则

启动阶段至少要保持三类版本一致：

```text
manifest.payloadVersion
= 本地 payload 目录记录的版本
= 文件清单对应的版本
```

如果本地已有 payload，但版本或文件清单与 manifest 不一致，应进入 payload 修复 / 重下流程。

不应把不同版本文件混在同一个可 handoff payload 中。

### 5. required 文件规则

manifest 中应区分：

```text
required = true
required = false
```

启动阶段只要求 handoff 必需文件全部通过。

例如必需项通常包括：

```text
本地主入口
菜单入口或菜单定义
handoff 后需要加载的最小脚本
```

非 handoff 必需的内容不应阻塞启动阶段，除非它会影响主菜单显示和后续任务入口。

### 6. payload 修复触发条件

以下情况应触发 payload 修复或重下：

```text
manifest 下载失败
manifest 解析失败
schemaVersion 不支持
platform / architecture 不匹配
required 文件缺失
required 文件 hash 不匹配
mainEntry 不存在
menuEntry 不存在
payloadVersion 不一致
```

修复次数由启动状态中的：

```text
payloadRepairCount
```

控制，不能无限重下。

### 7. manifest 与网络边界

启动阶段 manifest 只检查项目自己的分发源。

当前边界是：

```text
get.dingjiai.com
manifest
payload 文件
hash 校验
```

不在这里检查：

```text
GitHub
GitHub Releases
raw.githubusercontent.com
npm registry
winget source
Anthropic 下载源
```

这些属于后续安装阶段的 source preflight checkpoint 或组件 checkpoint。

### 8. 可 handoff 的判定

payload 可 handoff 必须同时满足：

```text
manifest 已下载并解析
schemaVersion 受支持
payloadVersion 明确
platform / architecture 匹配
required 文件全部存在
required 文件 hash 全部通过
mainEntry 存在
menuEntry 存在
handoff 参数可以完整组装
```

只有达到这个状态，启动阶段才允许进入：

```text
stage = handoff
```

## 十六、字段级深挖：handoffAccepted 写回机制与超时判断

本节把启动阶段唯一成功标准继续拆细。

本项目不能把 `Start-Process -Verb RunAs` 成功当作启动成功。

真正成功必须是：

```text
新管理员 cmd.exe 已进入本地主入口，并写回 handoffAccepted = true。
```

因此 `handoffAccepted` 是原始 bootstrap 和新管理员 CMD 之间的接管确认契约。

### 1. 写回责任

`handoffAccepted` 只能由新管理员 CMD 中的本地主入口写回。

不能由原始 PowerShell 写回。

原因是：

```text
原始 PowerShell 只能证明自己发起了 handoff；
只有新 CMD 主入口才能证明自己已经接管。
```

### 2. 写回条件

新管理员 CMD 写回 `handoffAccepted = true` 前，必须确认：

```text
当前进程运行在 cmd.exe 主宿主链路中
当前权限是管理员
startupId 与传入参数一致
workspaceRoot 可访问
payloadRoot 可访问
mainEntryPath 是当前入口或已成功进入当前入口
statePath 可写
logPath 可写
主菜单入口可达
```

只有这些条件成立，才允许写：

```json
{
  "stage": "completed",
  "handoffAccepted": true
}
```

### 3. 原始 PowerShell 等待逻辑

原始 PowerShell 发起 handoff 后进入短等待。

它只等待两种结果：

```text
看到 handoffAccepted = true，且接管写回字段与本次启动会话一致
或等待超时
```

原始 PowerShell 接受成功态时，必须同时校验 `acceptedWorkspaceRoot`、`acceptedStatePath`、`acceptedLogPath`、`acceptedPayloadRoot`、`acceptedMainEntryPath`、`acceptedSource` 和 `acceptedHandoffMode`。只看到 `stage = completed` 与 `handoffAccepted = true` 不足以判定接管成功。

它不继续做业务逻辑，不显示正式菜单。

推荐状态流：

```text
写 handoffAttempted = true
-> 发起管理员 CMD
-> 轮询 statePath
-> 看到 handoffAccepted = true
-> 原始 PowerShell 退出
```

如果超时：

```text
写 stage = failed
写 lastFailureStage = handoff
写 lastFailureCode = handoff-timeout
显示极简失败提示
退出
```

### 4. 超时不是重试许可

handoff 超时不应自动再次弹 UAC。

原因是：

```text
handoffAttempted 上限是 1 次；
重复弹 UAC 会造成用户困惑，也可能形成循环。
```

超时后应明确失败，把诊断信息留在日志中。

### 5. handoffAccepted 的最小状态示例

成功写回示例：

```json
{
  "startupId": "20260424-xxxx",
  "stage": "completed",
  "handoffAttempted": true,
  "handoffAccepted": true,
  "handoffAcceptedAt": "2026-04-24T10:00:00Z",
  "handoffMode": "admin-cmd",
  "acceptedBy": "local-main-entry"
}
```

失败示例：

```json
{
  "startupId": "20260424-xxxx",
  "stage": "failed",
  "handoffAttempted": true,
  "handoffAccepted": false,
  "lastFailureStage": "handoff",
  "lastFailureCode": "handoff-timeout",
  "lastFailureMessage": "Administrator cmd handoff did not report acceptance before timeout."
}
```

### 6. 不把进程存在当成功

以下信号都不能单独算成功：

- `Start-Process` 没有抛错。
- 出现了新的 `cmd.exe` 进程。
- 用户点击了 UAC “是”。
- 原始 PowerShell 没有立即退出。
- 日志文件被创建。

这些只能作为诊断信息。

成功信号只有：

```text
handoffAccepted = true
```

### 7. 新 CMD 写回后才显示主菜单

新管理员 CMD 应在确认自己可以写回状态后，再进入正式主菜单。

推荐顺序：

```text
解析 handoff 参数
-> 验证 workspace / payload / state / log
-> 确认管理员权限
-> 写回 handoffAccepted = true
-> 显示主菜单
```

这样原始 PowerShell 的退出和新 CMD 的菜单显示之间有清晰交接点。

### 8. 失败后的边界

如果新 CMD 无法写回 `handoffAccepted`，不应假装正常进入主菜单。

因为这会让原始 PowerShell 认为启动失败，而用户却看到另一个半接管窗口，状态会分裂。

因此：

```text
不能写回接管状态
= 不能进入正式主菜单
```

应该显示启动失败并退出，或停留在极简诊断提示。

### 9. handoffAccepted 防伪与并发保护

`handoffAccepted` 不是普通进度字段，而是启动阶段的接管成功判据，因此要防止误写、并发覆盖和伪造成功。

最低规则是：

```text
startupId 必须在启动早期生成，并具备足够随机熵。
同一 startupId 只能对应一个启动会话。
同一 workspaceRoot 下同时只能有一个活跃 handoff 会话。
statePath 必须落在 workspaceRoot\state 下。
statePath 文件名必须绑定 startupId。
原始 PowerShell 只能写 handoffAttempted、失败和诊断字段，不能写 handoffAccepted = true。
```

新管理员 CMD 写回 `handoffAccepted = true` 前，必须重新校验：

```text
startupId 与状态文件一致
statePath 是本次会话的状态文件
payloadRoot 与已校验 manifest 记录一致
mainEntryPath 是当前正在运行的本地主入口
当前进程具备管理员权限
当前进程来自预期 handoffMode = admin-cmd
```

如果其中任一项不一致，应拒绝写回接管成功，并记录失败原因。

状态写入还应遵守：

```text
写入使用临时文件 + 原子替换，避免半写状态。
写入前重新读取最新状态，避免覆盖其他会话的失败结论。
如果状态已是 completed，不再重复写入。
如果状态已是 failed，除非失败来自同一新 CMD 的可解释前置检查，否则不改写成 completed。
```

并发启动时的原则是：

```text
不同 startupId 不得互相覆盖状态文件。
旧会话不得接受新会话的 handoffAccepted。
新会话不得复用旧会话的 handoffAttempted。
锁文件只能保护当前启动会话，不能成为成功判据。
```

如果检测到并发冲突，启动阶段应选择保守失败，而不是尝试合并两个启动流程。

### 10. 状态文件权限边界

启动状态文件不保存敏感配置，但它会影响启动成功判断，因此仍应收敛权限边界。

原则是：

```text
状态文件只放在项目工作区 state 目录。
不接受用户传入任意 statePath。
不把状态文件放到公共可写目录。
不通过环境变量展开决定 statePath。
不跟随可疑 junction / symlink / reparse point 写入状态。
```

如果无法确认状态目录归属或写入目标安全，应停止 handoff，而不是把接管状态写到未知位置。

### 11. 用户提示与日志边界

启动阶段要同时服务两类读者：

```text
小白用户：只需要知道发生了什么、下一步怎么做。
维护者：需要知道哪个阶段、哪个字段、哪个条件失败。
```

因此错误信息要分层。

| 场景 | 给用户看的提示 | 写入日志的内容 |
| --- | --- | --- |
| 入口宿主不支持 | 当前窗口不适合启动安装器，请换 PowerShell 后重试。 | shell 名称、PowerShell 版本、进程位数、收敛尝试次数 |
| 平台硬门槛失败 | 当前系统暂不支持这个安装器。 | OS 版本、架构、cmd.exe / RunAs / 写入能力检查结果 |
| 分发源不可达 | 暂时无法获取安装器文件，请稍后重试。 | manifestUrl、HTTP/TLS/DNS 错误、重试次数、是否尝试 last-known-good |
| payload 校验失败 | 安装器文件校验失败，已停止启动以保护安全。 | 文件名、期望 hash、实际 hash、manifest 版本、清理动作 |
| UAC 被取消 | 没有获得管理员权限，安装器无法继续。 | handoffAttempted、UAC 结果、超时时间 |
| handoff 超时 | 管理员窗口没有完成接管，请关闭窗口后重试。 | startupId、statePath、等待时长、最后一次读取状态 |
| 状态文件异常 | 启动状态异常，已停止以避免错误继续。 | 状态解析错误、路径校验结果、锁状态、冲突 startupId |

用户提示应短、明确、中文优先；日志可以详细、结构化、便于排查。

### 12. 统一失败文案模板

启动阶段失败时，用户侧文案建议固定为：

```text
启动安装器失败。

原因：<一句话原因>
建议：<一句话建议>
日志：<logPath>

你可以关闭当前窗口后重新运行安装命令。
```

约束：

```text
原因不堆技术字段。
建议只给一个优先动作。
日志路径必须可复制。
如果 logPath 不可用，则提示“日志未能写入”。
```

技术字段写入日志，不直接堆给小白用户。

### 13. 原始窗口到新 CMD 的过渡体验

原始 PowerShell 在发起 handoff 后，不应立即显示大量输出，也不应继续显示菜单。

推荐体验是：

```text
正在打开管理员窗口，请在弹出的 UAC 提示中选择“是”……
如果新的管理员窗口已经出现，请在新窗口中继续。
```

随后原始窗口只做短等待：

```text
看到 handoffAccepted = true -> 显示“已切换到管理员窗口”并退出。
等待超时 -> 显示统一失败文案。
用户取消 UAC -> 显示统一失败文案。
```

不能出现两个长期可交互窗口。handoff 成功后，原始窗口必须退出；handoff 失败后，原始窗口只能显示失败原因和日志位置，不能降级进入菜单。

### 14. 日志脱敏与保留边界

启动日志从第一天就应存在，但不能变成敏感信息收集器。

启动阶段日志允许记录：

```text
startupId
阶段名
时间戳
PowerShell 版本
进程位数
管理员状态
workspaceRoot / payloadRoot / statePath / logPath
manifestUrl
payloadVersion
文件 hash 校验结果
失败阶段和失败码
重试次数
是否使用 last-known-good
```

启动阶段日志禁止记录：

```text
API key
token
用户输入的模型服务 URL 凭证
完整环境变量
完整 PATH 原文
Cookie
Authorization header
私有 SSH key 路径内容
settings.json / claude.json 中的敏感字段值
```

如果后续某些字段必须写入日志，应先做脱敏：

```text
只保留前后少量字符
中间使用 ***
或只记录是否存在，不记录具体值
```

日志保留策略建议为：

```text
启动日志按 startupId 分文件。
保留最近若干次启动日志。
超过保留上限时清理最旧日志。
清理失败不阻止启动，但要记录一次诊断事件。
```

具体保留天数或数量可以在实现前定，但文档层面必须明确：日志要可排查，也要避免无限增长和敏感信息泄漏。

## 十七、字段级深挖：last-known-good payload 策略边界

本节讨论启动阶段最关键的容灾策略：远程 manifest 或 payload 暂时不可用时，是否允许使用上一次已经校验通过的本地 payload 继续 handoff。

这里的结论应保持谨慎：

```text
last-known-good 可以作为启动可用性的候选容灾机制，
但不能绕过完整性校验，
也不能让未知、残缺、未记录版本的 payload 接管管理员 CMD。
```

### 1. last-known-good 要解决的问题

它只解决一种问题：

```text
项目分发源临时不可达，
但本机已有一套之前完整校验通过的 payload。
```

例如：

```text
get.dingjiai.com 临时不可访问
manifest 下载超时
payload 远程下载失败
```

此时，如果本地存在可信缓存，可以考虑继续进入管理员 CMD 主窗口，让用户至少能看到主菜单、诊断信息或执行不依赖更新的任务。

### 2. 它不能解决的问题

last-known-good 不用于绕过安全失败。

以下情况不允许使用：

```text
本地 payload 从未校验通过
本地 payload 文件缺失
本地 payload hash 不匹配
本地 payload version 不明确
本地 payload 与记录的 manifest 文件清单不一致
mainEntry 缺失
menuEntry 缺失
handoff 参数无法完整组装
本地状态记录显示该 payload 上次接管失败且未修复
```

这些都不是“远程临时不可达”，而是本地可信性不足。

### 3. last-known-good 的最低记录

如果后续允许 last-known-good，本地至少要记录：

| 字段 | 作用 |
| --- | --- |
| `payloadVersion` | 上次校验通过的 payload 版本 |
| `manifestSnapshotPath` | 上次使用的 manifest 快照 |
| `verifiedAt` | 上次完整校验时间 |
| `hashAlgorithm` | 上次使用的 hash 算法 |
| `files` | 上次校验通过的文件清单 |
| `mainEntry` | 上次可 handoff 的主入口 |
| `menuEntry` | 上次可 handoff 的菜单入口 |
| `lastHandoffAccepted` | 上次是否成功接管 |

没有这些记录，就不应该称为 last-known-good。

### 4. 使用前必须重新本地校验

即使 payload 曾经校验通过，使用前也必须重新校验本地文件。

流程应是：

```text
远程 manifest / payload 获取失败
-> 查找 last-known-good 记录
-> 读取上次 manifest 快照
-> 重新校验本地 required 文件 hash
-> 确认 mainEntry / menuEntry 存在
-> 标记 usingLastKnownGood = true
-> 允许 handoff
```

不能因为“以前通过过”就直接跳过本地 hash 校验。

### 5. 用户提示边界

如果使用 last-known-good，应明确告诉用户这是缓存启动：

```text
当前无法连接安装器分发源，已使用本机上次验证通过的启动文件继续。
```

但不要让用户做复杂选择。

启动阶段仍然保持小白友好：

```text
能安全继续就继续；
不能安全继续就失败并给出简单建议。
```

### 6. 状态字段建议

如果支持 last-known-good，启动状态可增加：

```text
usingLastKnownGood
lastKnownGoodVersion
lastKnownGoodVerifiedAt
lastKnownGoodReason
```

示例：

```json
{
  "stage": "payload",
  "payloadVersion": "1.0.0",
  "usingLastKnownGood": true,
  "lastKnownGoodVersion": "1.0.0",
  "lastKnownGoodReason": "manifest-download-timeout"
}
```

### 7. 与更新一致性的关系

last-known-good 会牺牲一点“始终使用最新 payload”的一致性，换取“分发源短暂故障时仍可启动”的可用性。

因此它的边界必须是：

```text
只允许使用上次完整校验通过的 payload；
只作为远程获取失败时的容灾；
不允许使用残缺或来源不明的本地文件；
使用前仍然重新校验 hash；
日志中必须明确记录。
```

### 8. 当前文档定稿倾向

当前建议倾向是：

```text
v1 可以设计 last-known-good 的状态字段和目录结构，
但是否默认启用，可以在实现前再做一次确认。
```

原因是这个策略同时影响：

- 安全性。
- 可用性。
- 更新一致性。
- 故障恢复体验。

在最终实现前，应明确选择：

```text
严格在线模式：远程失败即启动失败。
缓存容灾模式：远程失败时允许可信 last-known-good 继续 handoff。
```

无论选择哪种，都不能允许未校验 payload 接管管理员 CMD。

### 9. v1 默认策略建议

当前文档层面的建议是：

```text
v1 先把 last-known-good 作为可设计能力写入字段和目录结构，
但默认启动策略先采用严格在线模式，
只有用户后续明确确认缓存容灾模式后，才把它作为默认行为启用。
```

原因是：

```text
严格在线模式更简单、更一致、更容易审计；
缓存容灾模式更可用，但会引入新鲜度、回滚、撤销和故障解释成本。
```

因此 v1 实现前必须再次确认：

```text
是否允许远程分发源失败时继续使用本地缓存 handoff。
```

在确认之前，文档只把 last-known-good 定义为受控候选机制，不把它写成已经默认启用的行为。

### 10. 新鲜度、回滚与撤销边界

如果后续启用缓存容灾模式，还必须同时定义以下边界：

| 边界 | 必须回答的问题 |
| --- | --- |
| 新鲜度 | 上次校验通过的 payload 最多可以被使用多久？ |
| 最大回退版本 | 当前远程版本失败时，允许回退到多旧的本地版本？ |
| 最小允许版本 | 是否存在低于某版本就绝不允许启动的安全门槛？ |
| 撤销列表 | 如果某个 payload 版本被撤销，本地缓存如何知道不能再用？ |
| 回滚窗口 | 用户是否能从新版本失败回退到上一个稳定版本？ |
| 用户提示 | 缓存启动时如何让用户知道当前不是最新 payload？ |

这些规则不应等实现时临时补丁化。

如果没有撤销或最小允许版本机制，缓存容灾模式就只能作为有限可用性策略，不能被描述成完整供应链安全策略。

### 11. last-known-good 与安全失败的优先级

当可用性和安全性冲突时，启动阶段应优先安全失败。

也就是说：

```text
远程不可达 + 本地缓存完整可信 -> 可以考虑容灾。
远程可达但远程声明本地版本不可接受 -> 不应继续使用本地缓存。
本地 hash 不匹配 -> 必须失败。
本地 manifest 快照缺失 -> 必须失败。
主入口不在 required 文件清单中 -> 必须失败。
```

last-known-good 不能成为绕过 manifest、hash、required 文件和 handoff 安全边界的后门。

## 十八、MAS 对标收口

本节用于把前面已经展开的启动阶段细节收敛成更接近 MAS 的实现指导：直线、确定、少分支、可审计。

当前原则是：

```text
不再继续扩张启动阶段机制；
而是把现有机制收口为字段主责、状态机、固定预算和 MAS 硬化清单。
```

### 1. 字段主责矩阵

启动阶段可以复查某些能力，但每个字段都必须有唯一主责阶段，避免实现时 A/B/C/D/E 重复判断后得出不同结论。

| 字段 / 能力 | 主责阶段 | 可复查阶段 | 禁止阶段 / 禁止用途 |
| --- | --- | --- | --- |
| 当前平台是否 Windows | A. 宿主归一化 | B | D/E 不再重新分支平台逻辑 |
| PowerShell 宿主类型 | A. 宿主归一化 | 无 | 不进入业务阶段后再修宿主 |
| PowerShell 最低版本 | A. 宿主归一化 | B | D/E 不再处理版本兼容 |
| PowerShell 运行时健康 | A. 宿主归一化 | B | 不把 runtime 异常拖到 payload 阶段 |
| 进程位数 | A. 宿主归一化 | E 只校验结果 | D 不负责位数收敛 |
| 目标系统架构 | A. 宿主归一化 | B | D/E 不重新选择架构路径 |
| `cmd.exe` 可用性 | B. 平台硬门槛 | E | A 不执行 handoff；D 不检查 UI 宿主 |
| `Start-Process -Verb RunAs` 可用性 | B. 平台硬门槛 | E | B 只检查能力，不发起 UAC |
| 新进程启动能力 | B. 平台硬门槛 | E | D 不负责进程管理 |
| HTTPS 请求能力 | A. 宿主归一化 | D 使用项目分发源验证 | 不在启动阶段检查 GitHub、npm、winget 等业务源 |
| `%LOCALAPPDATA%` 可用性 | C. 本地工作区 | B 可做硬门槛预检 | A 不创建正式工作区 |
| workspaceRoot 可写 | C. 本地工作区 | E | D 只使用 C 给出的目标位置 |
| logs/state/temp 目录 | C. 本地工作区 | E | D 不重新定义目录结构 |
| 会话锁 | C. 本地工作区 | E | 不能作为启动成功判据 |
| manifestUrl | D. payload 获取与校验 | 无 | 不接受用户输入拼接 |
| manifest 结构 | D. payload 获取与校验 | 无 | E 只读取 D 的结论 |
| manifest 真实性边界 | D. payload 获取与校验 | 无 | 不能用 hash 替代来源真实性 |
| payload 文件清单 | D. payload 获取与校验 | E 只校验主入口属于已通过清单 | A/B/C 不判断 payload 可信 |
| payload hash | D. payload 获取与校验 | E 只读结论 | E 不重新下载 payload |
| mainEntryPath | D. payload 获取与校验 | E 必须复查路径安全 | 不能来自用户输入、PATH、当前目录或未知文件 |
| menuEntryPath | D. payload 获取与校验 | E 可复查存在性 | 菜单入口不能绕过 required 文件清单 |
| handoff 参数 | E. handoff 到管理员 CMD | 无 | D 不组装最终 UAC 命令 |
| handoffAttempted | E. handoff 到管理员 CMD | 原始 PowerShell 可读 | A/B/C/D 不写该字段 |
| handoffAccepted | E. handoff 到管理员 CMD | 原始 PowerShell 只读 | 原始 PowerShell 不能写 true |
| 用户业务组件状态 | 不属于启动阶段 | 安装阶段 checkpoint | 启动阶段禁止检查 Git、Claude、winget、npm、PATH、settings |

### 2. 启动状态机转移表

启动阶段的 stage 必须形成单向、有限、可审计的状态机。

允许转移：

| 当前 stage | 允许转移到 | 含义 |
| --- | --- | --- |
| `host-normalize` | `platform-gate` | 宿主已收敛或确认可继续 |
| `platform-gate` | `workspace` | 平台硬门槛通过 |
| `workspace` | `payload` | 本地工作区、日志、状态和锁已准备好 |
| `payload` | `handoff` | payload 已完整校验并可 handoff |
| `handoff` | `completed` | 新管理员 CMD 写回 `handoffAccepted = true` |
| 任意未完成 stage | `failed` | 当前阶段明确失败 |

禁止转移：

| 禁止转移 | 原因 |
| --- | --- |
| `completed` -> 任意 stage | 启动已完成，不能回滚重跑 |
| `failed` -> `completed` | 失败状态不能被伪造成功覆盖 |
| `handoff` -> `payload` | handoff 后不能回到下载阶段重写 payload |
| `payload` -> `workspace` | payload 阶段不能倒退重建工作区 |
| `workspace` -> `host-normalize` | 工作区建立后不再重新做宿主自重启 |
| 原始 PowerShell -> `completed` | 原始窗口不是接管方，不能写成功 |
| 新 CMD 未验证入口 -> `completed` | 未证明进入本地主入口，不能算成功 |

状态机原则：

```text
向前推进；
失败收口；
完成封口；
不允许隐式回跳；
所有回环都必须通过 attempt 字段受控表达。
```

### 3. 超时、重试与总预算常量表

MAS 风格更偏向确定规则，而不是开放区间。启动阶段应把所有回环定值化。

当前数值作为 v1 初始常量。项目上线并大规模测试后，可以基于真实失败率、慢网环境和用户机器性能再微调。

| 项目 | v1 初始值 | 说明 |
| --- | --- | --- |
| 宿主归一化尝试 | 1 次 | 重启到正确宿主后仍失败则退出 |
| 位数收敛尝试 | 1 次 | 64 位系统上从 32 位进程收敛到 64 位进程 |
| PowerShell runtime 健康检查重试 | 0 次 | 运行时不健康应早失败，不盲目重试 |
| workspace 创建重试 | 1 次 | 可清理明显残缺临时目录后再试一次 |
| workspace 单次准备预算 | 10 秒 | 包括创建目录、状态基座和日志基座 |
| manifest 下载重试 | 2 次 | 只针对项目分发源，不检查业务源 |
| manifest 单次请求超时 | 15 秒 | 超时后进入下一次 manifest 下载尝试 |
| payload 文件下载重试 | 2 次 | 单文件失败可有限重下 |
| payload 单文件请求超时 | 30 秒 | 超时后进入下一次 payload 文件下载尝试 |
| payload 修复重建 | 1 次 | 清理残缺 payload 后重建一次 |
| hash 校验失败重试 | 0 次 | hash 不匹配应清理并回到受控下载，而不是忽略 |
| UAC handoff 尝试 | 1 次 | 防止重复弹 UAC 和无限 handoff |
| handoffAccepted 等待 | 30 秒 | 原始 PowerShell 只短等待，不长期驻留 |
| 总启动预算 | 180 秒 | 超过总预算后进入 failed |

总规则：

```text
每个局部回环有上限；
整个启动阶段也要有总预算；
局部重试不能突破全局预算；
超出预算必须失败并写入 lastFailureStage / lastFailureCode。
```

### 4. handoffAccepted 的最小语义

`handoffAccepted` 必须保留，但主流程只赋予它一个最小、单一的语义：

```text
handoffAccepted = true 只表示：
新管理员 cmd.exe 已进入已校验 payload 的本地主入口，
并确认自己接管后续主菜单流程。
```

它不表示：

```text
安装已经成功；
业务组件已经健康；
payload 永远可信；
网络已经可用；
后续任务不会失败。
```

防伪、并发、原子写入、路径安全属于 `handoffAccepted` 的安全边界，不改变它的核心业务语义。

### 5. MAS 启动硬化补齐清单

基于 MAS 对标审阅，当前启动阶段还应补齐以下硬化点。

#### 5.1 环境基线收敛

启动早期不仅要检查环境，还要先把关键执行环境收敛到可控形态。

原则：

```text
先收敛环境基线；
再做能力检查；
再进入 payload 和 handoff。
```

包括：

- 不依赖用户当前目录定位主入口。
- 不依赖用户 PATH 搜索关键入口文件。
- 核心系统工具应通过可信系统位置或能力检查确认。
- 不把用户 shell 的临时变量当作长期状态源。
- 不让污染的环境变量决定 payload、state、logs 的位置。

#### 5.2 PowerShell runtime 健康门槛

入口宿主检查不应只看 PowerShell 版本，还要确认运行时足够健康。

建议字段包括：

| 字段 | 目的 |
| --- | --- |
| PowerShell 版本 | 确认最低语法和命令能力 |
| LanguageMode | 避免受限语言模式阻断启动脚本 |
| HTTPS 请求能力 | 确认能访问项目分发源 |
| 基本文件写入能力 | 确认能准备工作区前置条件 |
| 新进程启动能力 | 确认后续 handoff 有基础能力 |
| 错误捕获能力 | 确认失败能被记录和展示 |

如果 runtime 不健康，应早失败并给出简单用户提示，而不是拖到 payload 阶段暴露成复杂错误。

#### 5.3 终端兼容策略

原始入口可能来自 Windows Terminal、PowerShell 5.1、PowerShell 7、传统控制台或其他宿主。

文档应明确：

```text
可继续的宿主：满足最低能力且能完成 bootstrap 的 PowerShell 宿主。
需收敛的宿主：位数不正确、宿主行为不稳定但可重启到系统 PowerShell 的入口。
应失败的宿主：无法执行必要 PowerShell、无法发起新进程、无法写入工作区或无法发起 UAC 的入口。
```

宿主兼容策略的目标不是支持所有外壳，而是把入口收敛到可控 bootstrap 形态。

#### 5.4 系统架构矩阵

启动阶段需要显式记录系统架构处理策略，避免只覆盖 x86/x64。

| 系统 / 进程形态 | 启动策略 |
| --- | --- |
| 64 位 Windows + 64 位 PowerShell | 可继续 |
| 64 位 Windows + 32 位 PowerShell | 尝试一次收敛到 64 位系统 PowerShell |
| 32 位 Windows | 不支持或单独声明支持边界 |
| ARM64 Windows | 实现前单独确认支持策略 |
| 非 Windows | Windows 启动入口直接失败 |

未明确支持的架构不能假装支持，应早失败并记录诊断。

#### 5.5 入口落地形态防呆

启动阶段应识别明显错误或不稳定的入口形态。

应拒绝或收敛：

```text
从不可信临时位置执行本地主入口；
从压缩包内部直接运行；
从 UNC / 网络路径运行主入口；
从 workspaceRoot 外部伪造 payload 主入口；
通过相对路径、当前目录或 PATH 命中主入口；
通过 junction / symlink / reparse point 跳到允许目录外。
```

远程一行命令入口可以保持极薄，但本地主入口一旦进入 payload / handoff 链路，就必须满足可审计的落地形态。

### 6. MAS 对标后的收口结论

本项目启动阶段与 MAS 的关系应固定为：

```text
骨架直接对标 MAS：极薄入口、宿主收敛、权限收敛、早失败、重入保护、稳定主宿主接管。
实现保持 clean-room：学习机制，不复制 GPL-3.0 代码。
业务合理偏离：增加 payload / manifest / handoffAccepted，以适配本项目安装器架构。
文档继续瘦身：用字段主责矩阵、状态机和常量表减少重复解释。
```

也就是说，后续不再发明另一套启动机制；只继续把这一套 MAS 对标机制定值化、瘦身化、可实现化。

## 十九、当前已经明确的共识

基于本轮讨论，当前已明确的共识有：

1. 启动阶段必须强借鉴 MAS。
2. 借鉴重点是**启动兼容性处理方式**，不是激活业务代码。
3. 本地工作区路径可以固定。
4. 当前建议的固定路径为：

```text
%LOCALAPPDATA%\dingjiai-installer
```

5. 启动阶段的唯一成功标准是：

> **新的管理员 `cmd.exe` 主窗口成功启动并接管安装器主入口。**

6. 如果关键检查未通过，就算启动失败。
7. 为了提升兼容性，启动阶段必须允许宿主归一化、payload 修复等受控回环。
8. 五段式边界已经固定：A 修正宿主，B 只做平台能力门槛，C 准备工作区，D 校验 payload，E 才执行管理员 CMD handoff。
9. `manifest + hash` 首先证明完整性；真实性必须来自项目认可的分发源、签名、信任锚或后续发布安全机制。
10. `handoffAccepted = true` 只能由新管理员 CMD 中的本地主入口写回，且写回前必须验证 startupId、statePath、payloadRoot、mainEntryPath、管理员权限和 handoffMode。
11. handoff 参数必须白名单化、路径必须绝对化并限制在允许目录中，不能通过用户输入或字符串拼接决定主入口。
12. 状态写入要防止伪造和并发覆盖；原始 PowerShell 只能等待和读取接管结果，不能伪造成功。
13. 用户提示和日志必须分层：用户看到简单中文原因和建议，日志记录结构化诊断字段。
14. 启动日志必须脱敏，不记录 API key、token、完整环境变量、授权头或用户配置中的敏感值。
15. last-known-good 当前只作为受控候选机制；v1 默认策略建议先严格在线，是否启用缓存容灾需要实现前再次确认。

## 二十、下一步讨论点

当前文档已经把启动阶段从总体框架推进到字段级契约，但在定稿前仍建议继续用审阅视角打磨以下问题：
