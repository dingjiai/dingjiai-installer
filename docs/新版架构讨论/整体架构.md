# 整体架构

本文整理当前这轮关于“彻底重做安装器架构”的讨论结果。

目标不是立即写代码，而是先把整体方向、分层边界、循环结构、启动阶段原则和后续拆分方式明确下来。后续应当采用“**讨论一点，写一点**”的方式，持续完善这些架构文档，再进入实现阶段。

---

## 一、当前重构方向

当前讨论的方向不是在现有仓库上做零碎修补，而是：

> **在充分借鉴 MAS 架构与产品形态的前提下，重新定义一个适合本项目的全新安装器骨架。**

这里的“借鉴 MAS”指的是借鉴：

- 单行入口分发
- 菜单驱动交互
- 环境与权限先收敛再进入主逻辑
- 启动兼容性处理方式
- 分阶段、可审计的脚本流程
- 开发期可拆分、发布期可合并的组织思路

但不意味着直接继承其现成代码或业务逻辑。

---

## 二、为什么启动阶段要强借鉴 MAS

用户的核心判断是：

- `massgravel/Microsoft-Activation-Scripts` 经过多年迭代
- 兼容性与启动流程成熟
- 除了具体业务（MAS 做激活，本项目做安装与下载）外，其余大量架构思路都值得学习
- 尤其是启动阶段，涉及宿主、权限、重入、位数、UAC、远程执行和 handoff，不能重复造轮子

当前讨论形成的共识是：

> **本项目启动阶段应当把 MAS 的成熟机制作为默认参考模型，优先复用其产品流程和兼容性收敛思路，而不是自行发明一套未经验证的启动机制。**

同时必须明确：

> **强借鉴 MAS 的机制，不等于复制 MAS 的 GPL-3.0 代码。**

本项目应当学习 MAS 的产品哲学、流程模型和兼容性处理方式，并用 MIT 兼容的新代码重新实现。

MAS 最值得学习的不是某个单独脚本，而是这一套稳定方法。

当前已确认：启动阶段后续就按照这个方向继续细化，不再另起一套启动机制。

定稿方向是：

```text
启动阶段严格参考 MAS 的成熟启动机制；
除非 MAS 的做法和本项目“payload + 管理员 CMD 主窗口”目标冲突，
否则不自行发明新机制。
```

具体落地时应按 MAS 的成熟模式拆解并映射到本项目：

```text
极薄入口
-> 宿主归一化
-> 位数收敛
-> 权限收敛
-> 重入保护
-> 平台硬门槛尽早失败
-> payload 获取与校验
-> 管理员 CMD handoff
```

实现要求仍然是：

```text
学习机制
重新实现
不复制代码
保持 MIT 兼容
```

MAS 的稳定方法包括：

1. 先把当前宿主修正到正确形态。
2. 先解决权限，再进入主功能。
3. 先卡住最低平台门槛。
4. 用“重启自身”的方式做兼容性收敛。
5. 用重入保护防止无限循环。

因此，本项目最适合继承的核心原则是：

> **先归一化运行环境，再进入主逻辑。**

## 三、不能直接“删掉激活业务后继承 MAS 代码”的原因

虽然 MAS 的骨架非常值得学习，但当前也已经明确：

### 1. 法律与许可证层面

MAS 使用 GPL-3.0。

这意味着：

- 可以学习其架构与实现思路
- 可以模仿产品形态
- 可以重写同类机制
- 但不适合简单复制、裁剪其代码后继续当作当前 MIT 项目的代码基础

### 3. 当前项目的 MIT 定位

当前项目既有定位是 MIT 开源。

因此，如果直接复制、裁剪或派生 MAS 的 GPL-3.0 代码，会影响当前项目的许可证策略。

这也是为什么当前选择是：

> **学习 MAS 的产品哲学和流程模型，但重新写一个许可清晰的新骨架。**

### 4. MAS 仓库结构观察

本轮实际查看 MAS GitHub 仓库后，确认其顶层结构非常薄：

```text
README.md
LICENSE
MAS/
```

`MAS/` 下面主要分为：

```text
MAS/All-In-One-Version-KL/MAS_AIO.cmd
MAS/Separate-Files-Version/Activators/*
MAS/Separate-Files-Version/Change_Office_Edition.cmd
MAS/Separate-Files-Version/Change_Windows_Edition.cmd
MAS/Separate-Files-Version/Check_Activation_Status.cmd
MAS/Separate-Files-Version/Extract_OEM_Folder.cmd
MAS/Separate-Files-Version/Troubleshoot.cmd
```

这说明 MAS 的“通用骨架”并不是一个天然独立库，而是和业务脚本长期融合在一起。

因此，删除激活业务后会剩下很少内容，这反而进一步说明：

> **真正值得继承的是组织原则与流程模型，而不是文件本身。**

### 5. MAS 的双入口分发方式

MAS README 中体现了两类入口：

1. PowerShell 远程执行入口
2. 传统下载入口，即下载 `MAS_AIO.cmd` 或 zip 后手动运行

这说明 MAS 并不只依赖一种启动方式，而是保留了更稳妥的容灾式分发路径。

本项目后续也可以考虑类似策略：

- v1 主推远程一行命令
- 保留传统离线 / 手动下载入口作为容灾路径
- 避免用户只要远程入口失败就完全无法启动

### 6. AIO 分发 + Separate 开发

MAS 很重要的工程形态是：

```text
开发期 Separate Files
发布期 All-In-One / 极简入口
```

这对本项目非常有参考价值。

后续应当区分三种结构：

1. **source layout**
   - 开发期源码结构
   - 便于维护、审计、测试
2. **published payload layout**
   - 实际发布到 `get.dingjiai.com` 的 payload 结构
   - 便于 bootstrap 拉取和校验
3. **optional AIO artifact**
   - 可选的一体化分发产物
   - 便于用户手动下载、离线运行、传播

这能避免把“源码目录结构”和“用户下载结构”混为一谈。

## 四、整体产品运行架构

当前讨论认为，整个产品运行架构应当是：

```text
用户复制一行命令
-> 原始 PowerShell / Terminal 只负责引导
-> Remote bootstrap
-> 最小启动检查
-> 下载或刷新本地 payload
-> 打开新的管理员 CMD 主窗口
-> 进入主菜单
-> 选择安装 / 更新 / 卸载
-> 进入对应任务编排器
-> 每个组件按 checkpoint 执行
-> 写入状态 / 日志 / 结果
-> 返回主菜单
```

这里要特别强调三个认知：

1. **原始窗口不是主程序，只是引导壳。**
2. **新的管理员 `cmd.exe` 才是真正的稳定长期宿主。**
3. **所有业务都必须经过 checkpoint 流程，而不是直接“下载并安装”。**

---

## 五、整体分层架构

当前建议的总分层如下：

### 1. 用户入口层

负责：

- 用户复制一行命令
- 在原始 PowerShell / Terminal 中触发 bootstrap

特点：

- 只负责进入系统
- 不是长期 UI 宿主

### 2. Bootstrap 层

负责：

- 远程 bootstrap
- 最小启动检查
- 下载 / 刷新 payload
- 切换到管理员 CMD

特点：

- 保持极薄
- 不进入业务安装逻辑

### 3. 稳定交互宿主层

负责：

- 管理员 `cmd.exe` 主窗口
- 长期控制台交互宿主
- 后续所有菜单和任务执行的运行环境

特点：

- 这是 Windows 主战场
- 一旦 handoff 成功，后续都应留在这里

### 4. 菜单层

负责：

- 显示主菜单
- 接收编号输入
- 路由到安装 / 更新 / 卸载
- 完成后返回主菜单

特点：

- 只做导航
- 不写重逻辑

### 5. 任务编排层

负责：

- 安装编排器
- 更新编排器
- 卸载编排器

特点：

- 三者并列
- 共享状态、日志和组件规则
- 但流程独立

### 6. 组件 checkpoint 层

这是后续真正的执行内核。

每个组件都必须走统一模型：

```text
discovery -> allowance -> action -> new-shell validation -> component config -> final re-validation
```

特点：

- 强调审计性
- 强调可解释性
- 强调每个组件独立闭环

### 7. 状态与结果层

负责：

- 本地缓存
- 状态文件
- 运行日志
- 最终结果

特点：

- 是底层横切层
- 从第一天就应该存在
- 不是后期补丁

## 六、主循环结构

当前讨论已经明确：整体流程里至少存在 **3 层循环**。

### 1. 主菜单循环

```text
显示主菜单 -> 用户选择任务 -> 进入任务 -> 任务结束 -> 返回主菜单
```

这是外层总循环。

### 2. 任务内组件循环

```text
生成组件清单 / 执行计划
-> 取下一个组件
-> 该组件完成
-> 再取下一个组件
-> 全部组件完成
-> 最终验证
```

这是中层循环。

### 3. 单组件 checkpoint 循环

```text
discovery
-> allowance
-> action
-> validation
-> 若不健康则修复 / 重试 / 路径切换
-> 再 validation
-> 直到组件完成或明确失败
```

这是内层循环，也是安装器真正的核心。

## 八、启动阶段在整体架构中的位置

当前讨论中，启动阶段被明确为：

> **只负责把主战场准备好，不负责打仗。**

也就是说，启动阶段不做业务安装，而只完成以下内容：

1. 确认机器可以安全进入主流程
2. 准备本地 launcher payload
3. 成功切换到新的管理员 `cmd.exe` 主窗口
4. 把控制权交给主菜单

启动阶段的唯一成功标准是：

> **新的管理员 `cmd.exe` 主窗口已成功启动，并接管安装器主入口。**

这一点已经被明确为启动阶段的唯一判断标准。

## 九、启动阶段的五段式结构

当前讨论建议把启动阶段固定为五段：

```text
A. 宿主归一化
B. 平台硬门槛
C. 本地工作区
D. payload 获取与校验
E. handoff 到管理员 CMD
```

其中：

- **A 和 E 最像 MAS 的成熟实践**
- **D 是本项目比 MAS 更需要重点设计的部分**

### A. 宿主归一化

目标：先把 bootstrap 当前运行壳收敛到正确、可控的形态。

检查内容包括：

- Windows
- PowerShell 宿主
- PowerShell 最低版本
- 当前进程位数是否正确
- HTTPS 下载能力
- 文件写入能力
- 启动新进程能力
- 发起提权能力

处理原则：

```text
宿主不正确
-> 重启到正确宿主
-> 重新进入启动流程
-> 若仍不正确，则退出
```

### B. 平台硬门槛

只检查不能继续的硬门槛，例如：

- Windows 版本
- 系统架构
- `cmd.exe` 可用性
- `Start-Process -Verb RunAs` 可用性
- 本地工作区可写性
- 临时目录可写性

原则：

> **不支持就早点停，不把用户拖进后面一定会炸的流程。**

### C. 本地工作区

当前已确认：

```text
%LOCALAPPDATA%\dingjiai-installer
```

建议至少包含：

- `cache`
- `payload`
- `logs`
- `state`
- `temp`

同时要处理：

- 旧 payload 是否存在
- 是否完整
- 是否残缺
- 是否存在会话占用
- 是否需要清理后重建

### D. payload 获取与校验

这一段重点不是“下载成功”，而是：

> **拿到完整、可信、可 handoff 的本地主程序。**

当前建议的思路是：

- 先拉 manifest
- 再下载 payload 文件
- 再校验完整性
- 失败时有限重试

当前也讨论了 `manifest + hash` 的思路，其含义是：

- 不能只看“下载成功没有”
- 必须确认“本地这套 payload 是完整且可信的”

启动阶段的网络硬门槛应只围绕项目自己的分发源：

- 当前宿主具备 HTTPS 请求能力
- DNS / TLS 能支持访问 `get.dingjiai.com`
- manifest 可下载、可解析
- payload 可下载
- payload hash 校验通过

外网可达性、GitHub 可达性可以作为诊断信息记录，但不应阻止进入管理员 CMD 主菜单。

GitHub、GitHub Releases、`raw.githubusercontent.com`、winget、npm 等业务下载源，应放到安装阶段的 source preflight checkpoint 或各组件 checkpoint 中检查，而不是塞进启动阶段。

### E. handoff 到管理员 CMD

这是启动阶段的最终判定点。

需要确认：

- 本地主入口文件存在
- 主菜单入口存在
- handoff 参数完整
- 管理员 `cmd.exe` 能启动
- 新窗口启动后真正接管本地主入口

成功标准已明确为：

> **新的管理员 `cmd.exe` 主窗口成功启动并接管安装器主入口。**

## 十、启动阶段里的 3 个回环

按 MAS 风格，启动阶段至少应有 3 个受控回环：

### 1. 宿主归一化回环

```text
宿主不对
-> 自重启到正确宿主
-> 重新进入启动流程
```

### 2. payload 修复回环

```text
本地 payload 残缺 / 远程下载失败
-> 清理残缺内容
-> 重下
-> 再校验
```

### 3. handoff 单次重入保护

```text
准备 handoff
-> 启动管理员 CMD
-> 若失败则退出
-> 不允许无限 handoff
```

这三个回环分别对应：

- 兼容性收敛
- 本地主程序修复
- 交棒保护

兼容性收敛是启动阶段的核心原则之一。

启动阶段不应该在错误宿主、错误位数、错误权限或脏 payload 状态下继续硬跑，而应该采用 MAS 风格的方式：

```text
发现当前运行形态不适合继续
-> 切换到更正确的运行形态
-> 带状态标记重新进入启动流程
-> 重新检查
-> 通过后继续
-> 仍不通过则明确失败退出
```

当前建议的稳定目标是：

```text
64 位 Windows
+ 可用 PowerShell bootstrap 宿主
+ 可写本地工作区
+ 完整可信 payload
+ 新开的管理员 cmd.exe 主窗口
```

所有收敛都必须有限，并且必须有状态标记，避免无限重启、无限重下或无限 handoff。

启动阶段应使用独立的启动状态文件记录本次启动会话，例如：

```text
%LOCALAPPDATA%\dingjiai-installer\state\startup-<startupId>.json
```

该状态文件只记录启动阶段状态，例如：

- 当前阶段：宿主归一化、平台门槛、本地工作区、payload、handoff、完成、失败
- 是否已经尝试宿主归一化
- payload 修复 / 重下次数
- 是否已经尝试管理员 CMD handoff
- 新管理员 CMD 是否已写回接管标记
- 上一次失败阶段和失败原因

它不记录 Git、Claude、gh、rg、Python 等业务组件状态，也不记录用户 API key 或产品配置。

## 十一、启动前置能力

“基础能力”这个说法过于宽泛，当前建议改成：

**启动前置能力**

只包含：

1. HTTPS 下载能力
2. 文件写入能力
3. 新进程启动能力
4. 管理员提权能力
5. 远程 payload 完整性校验能力

这组能力定义只服务于启动阶段，不外扩到后续业务安装。

## 十二、启动阶段明确不做什么

当前已经明确，启动阶段必须严格控肥。

启动阶段禁止做的事包括：

- 不发现 Git / Claude / gh 是否已安装
- 不判断业务组件版本是否过旧
- 不修 PATH
- 不写 `settings.json`
- 不下载业务组件
- 不显示正式功能菜单
- 不进入安装 / 更新 / 卸载逻辑

也就是说：

> **启动阶段只负责把主战场准备好，不负责打仗。**

## 十三、启动阶段字段级结论摘要

启动阶段的字段级设计已进一步明确，详细内容记录在 `docs/新版架构讨论/启动阶段（一次性）.md`。

`docs/新版架构讨论/启动阶段（一次性）.md` 是启动阶段的详细规范源；本文只保留启动阶段摘要、分层位置和跨层边界。如果两份文档出现细节差异，启动阶段的字段、顺序、状态、handoff、payload 和安全边界以启动阶段专题文档为准，本文应回写同步摘要，避免双写漂移。

当前总架构层面只保留以下摘要结论：

1. 入口宿主检查只覆盖启动宿主能力，不检查 Git、Claude、winget、GitHub、npm、PATH 或用户配置。
2. 64 位 Windows 上如果当前是 32 位 PowerShell 进程，应先尝试一次收敛到 64 位系统 PowerShell；尝试后仍失败则停止，避免无限重启。
3. 启动阶段五段式边界固定为：A 宿主归一化、B 平台硬门槛、C 本地工作区、D payload 获取与校验、E handoff 到管理员 CMD。
4. B 阶段只检查 `cmd.exe`、`Start-Process -Verb RunAs`、新进程启动等 handoff 能力；真正打开管理员 `cmd.exe`、写入 `handoffAttempted`、等待 `handoffAccepted` 只能发生在 E 阶段。
5. C 阶段只准备 workspace、state、logs、temp、锁和会话基座；payload 是否完整、可信、版本一致，只能由 D 阶段判断。
6. UAC handoff 的目标不是单纯拿到管理员权限，而是打开新的管理员 `cmd.exe`，进入本地 payload 主入口，并完成接管写回。
7. 启动状态文件只记录启动链路状态、回环次数、payload 状态、handoff 状态和失败原因，不记录业务组件状态或敏感配置。
8. manifest 必须声明 payload 版本、文件清单、hash、主入口和菜单入口；required 文件全部校验通过后才允许 handoff。
9. `manifest + hash` 首先证明完整性；真实性还需要项目认可的分发源、签名、信任锚或后续发布安全机制支撑。
10. `handoffAccepted = true` 只能由新管理员 CMD 中的本地主入口写回，原始 PowerShell 只能等待和读取，不能伪造成功。
11. `handoffAccepted` 写回前必须验证 startupId、statePath、payloadRoot、mainEntryPath、管理员权限和 handoffMode，并使用受控状态写入方式避免伪造和并发覆盖。
12. handoff 参数必须白名单化；路径必须规范化为绝对路径并限制在允许的 workspace / payload / logs / state 范围内；主入口不能来自用户输入、PATH 搜索、当前目录或未校验文件。
13. 用户提示和日志必须分层：用户看到简短中文原因、建议和日志位置；日志记录阶段、字段、期望值、实际值、重试次数和失败码。
14. 启动日志必须脱敏，不记录 API key、token、完整环境变量、授权头或用户配置中的敏感值，并应具备保留与清理边界。
15. last-known-good payload 只能作为远程分发源失败时的候选容灾路径，且使用前必须基于上次 manifest 快照重新校验本地 hash。
16. v1 默认策略建议先采用严格在线模式；last-known-good 可先设计字段和目录结构，但是否默认启用缓存容灾，需要实现前再次确认。
17. 启动阶段后续不再扩张新机制，而是按 MAS 对标收口：字段主责矩阵、状态机转移表、固定超时 / 重试 / 总预算常量表、MAS 启动硬化补齐清单。
18. 启动阶段常量采用 v1 初始值：宿主归一化 1 次、位数收敛 1 次、PowerShell runtime 健康检查重试 0 次、workspace 创建重试 1 次、manifest 下载重试 2 次、payload 文件下载重试 2 次、payload 修复重建 1 次、hash 校验失败重试 0 次、UAC handoff 尝试 1 次。
19. 启动阶段 v1 初始时间预算为：workspace 单次准备 10 秒、manifest 单次请求超时 15 秒、payload 单文件请求超时 30 秒、`handoffAccepted` 等待 30 秒、总启动预算 180 秒；上线大规模测试后可基于真实数据微调。
20. `handoffAccepted` 的主流程语义应保持最小化：只表示新管理员 `cmd.exe` 已进入已校验 payload 的本地主入口，并确认接管后续主菜单流程。
21. MAS 启动硬化补齐项包括环境基线收敛、PowerShell runtime 健康门槛、终端兼容策略、系统架构矩阵和入口落地形态防呆。

这些结论不改变整体分层，只是把启动阶段从“框架级原则”推进到“可实现的字段级契约”。

## 十四、当前明确共识

截至本轮讨论，当前已经明确的共识有：

1. 本项目准备以“整体重构”方式重新定义架构，而不是继续在现有原型上局部修补。
2. 必须强借鉴 MAS，但借鉴的是架构、流程、兼容性处理方式，不是其激活业务代码。
3. 原始 PowerShell / Terminal 只作为 bootstrap 宿主，不作为长期 UI 主宿主。
4. 新开的管理员 `cmd.exe` 是 Windows 路线下的真正稳定主窗口。
5. 安装 / 更新 / 卸载应当是并列编排器，而不是安装逻辑的附属分支。
6. 所有组件都必须 checkpoint 化。
7. 状态与日志应当从第一天就是正式层。
8. 启动阶段必须允许 MAS 风格的兼容性收敛与受控回环。
9. 本地工作区路径可以固定为：

```text
%LOCALAPPDATA%\dingjiai-installer
```

10. 启动阶段的唯一成功标准是：

> **新的管理员 `cmd.exe` 主窗口成功启动并接管安装器主入口。**
